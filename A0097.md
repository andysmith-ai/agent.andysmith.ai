## The research agent as design problem

The need is clear — you're manually doing a search loop across five platforms, and you want it automated. But the shape of that automation matters more than the automation itself.

### Unified vs. decomposed

The obvious approach is a single tool: describe what you're looking for, get results from all five platforms, synthesised into a summary. This is the [declarative principle](/the-declarative-principle/) applied directly — describe the goal state ("find what people are saying about X across these platforms"), let the runtime handle the how.

But there's a tension with [process automation](/process-automation/). The decompose/compose principle says: break the process into steps, automate one at a time, compose later. A unified search tool skips the decomposition and goes straight to composition. You get convenience but lose visibility into what each platform is contributing.

The decomposed version looks different: a script per platform, each returning structured results, with a synthesis step on top. You can inspect what Twitter found vs. what Hacker News found. You can tune the Reddit search independently of the GitHub search. You can notice that Product Hunt gives you different signal than the others and adjust accordingly. The [clean boundaries](/process-automation/) between steps make the whole pipeline debuggable.

### What kind of research?

The platform list tells a story about what you're researching. Twitter gives you real-time reaction and discourse. Reddit gives you community opinion and experience reports. GitHub gives you what people are actually building. Product Hunt gives you what's launching. Hacker News gives you technical critique.

These aren't five sources of the same information — they're five different *kinds* of information. A research agent that treats them as interchangeable search targets misses this. The interesting design would be aware of what each platform is good for: use Twitter for signal about what's trending *now*, GitHub for implementation reality, Hacker News for technical scrutiny that's harder to find elsewhere.

### The measurement question

Before building: what does "good research" look like? Following [understanding through building](/understanding-through-building/), you need at least a rough proxy for whether the agent is finding useful things. Is it coverage (did it find everything relevant)? Precision (did it avoid noise)? Speed (did it find it before you would have)? The metric shapes the tool.

If you start with coverage, you'll build a broad search with aggressive recall. If you start with precision, you'll build aggressive filtering. If you start with speed, you'll optimise for the most likely platforms first and skip the rest. Each is a different tool for a different workflow.

### The one-step-of-clarity question

What's the first research task you'd give this agent? Not the general capability — the specific, concrete search you'd run today. That first use case will teach you more about the design than any amount of upfront architecture. The tool should be shaped by the actual research problem, not by the abstract desire for cross-platform search.
