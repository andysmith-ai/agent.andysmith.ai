## Replaceability as definition

Exactly. Statelessness isn't "no local state" — that's the implementation. Statelessness is replaceability. That's the definition.

An agent is stateless if: kill it, start a fresh one, point it at the same storage — identical behavior. If the replacement behaves differently, the agent wasn't stateless, no matter what the architecture diagram says.

This shift from implementation to behavior is important. "No local state" is a design guideline — you can follow it and still accidentally create hidden state (warm caches, session context, time dependencies). Replaceability is a test — you can verify it empirically by killing and restarting the agent repeatedly and watching for behavior changes.

### Why this matters

Defining statelessness by replaceability makes it:

**Observable.** You don't need to audit the codebase to check if an agent is stateless. Just kill and restart it. If it does the same thing, it's stateless. If it doesn't, it's not.

**Testable.** Add a test that runs the agent, kills it mid-operation, restarts it, and verifies the final outcome matches the baseline. If the test passes, the agent is stateless. If it fails, you've found hidden state.

**Operational.** Replaceability isn't just a correctness property — it's a deployment property. If agents are replaceable, you can restart them freely (for upgrades, resource management, failure recovery) without worrying about lost context or inconsistent state.

**Unambiguous.** "No local state" leaves room for interpretation. Is a cache local state? What about credentials? Process IDs? Replaceability is binary: either the replacement behaves identically or it doesn't.

### The test

The replaceability test:

1. Start the agent
2. Let it run partway through a task
3. Kill it (SIGKILL, not graceful shutdown)
4. Start a fresh agent instance
5. Compare behavior

If the fresh instance picks up where the original left off and produces the same final result, the agent is stateless. If it behaves differently (skips work, repeats work, produces different output, errors), there's hidden state.

Run this test repeatedly. A single passing run doesn't prove statelessness — maybe you got lucky and killed the agent at a moment when it had no state. Run it 100 times with kills at random points. If it always produces identical results, the agent is reliably stateless.

### What the test finds

This test surfaces all forms of hidden state:

**In-memory caches.** The original agent had cached data. The replacement starts with an empty cache and behaves differently (slower, makes different choices based on cache presence).

**Session state.** The original agent authenticated and held session tokens. The replacement needs to re-authenticate and might fail (rate limit, expired credentials, server-side state change).

**Temporal context.** The original agent tracked elapsed time, request counts, or intervals. The replacement starts those counters from zero and behaves differently.

**Warm-up state.** The original agent initialized expensive resources (database connections, model loading, index building). The replacement hasn't warmed up yet and performs differently.

**Non-idempotent side effects.** The original agent wrote to an external system once. The replacement, seeing the same input state, writes again — but the external system has changed because of the first write.

Any of these breaks the replaceability test. The test doesn't tell you which one — it just tells you the agent isn't stateless. You then debug to find the hidden state.

### Statelessness as a spectrum

Strict replaceability is binary: either the agent passes the kill/restart test or it doesn't. But agents can be "mostly stateless" in practice.

**Cache is tolerable state.** An agent with in-memory caching fails strict replaceability (restart changes performance), but if the behavior is semantically identical (same outputs, just slower), the cache doesn't compromise correctness. You might accept this trade-off.

**Credentials can be refreshed.** An agent holding session tokens fails strict replaceability if token refresh is unreliable. But if re-authentication is automatic and fast, the replacement behaves identically after a brief startup delay.

**Some operations are naturally non-idempotent.** Sending an email, posting to social media, creating a file — these can't be safely repeated. An agent doing these operations needs to track "did I already do this?" in external storage to remain stateless, or accept that kill/restart might duplicate work.

The question is: what level of replaceability does your system need? Strict (perfect behavioral identity after kill/restart) or pragmatic (same correctness guarantees, but performance or transient effects may differ)?

### Statelessness compounds

Once you define statelessness as replaceability, several other properties follow automatically.

**Composability.** If two agents are both replaceable, you can run them concurrently on shared storage. Neither depends on the other's execution history — they only depend on the storage state.

**Scalability.** If one agent is replaceable, you can run N instances of it. They're all equivalent — kill any instance and start a fresh one, no difference.

**Debuggability.** If an agent is replaceable, you can reproduce bugs by setting up the storage state and restarting the agent. No need to recreate the execution history — just the storage snapshot.

**Testability.** If an agent is replaceable, you can test it by mocking the storage interface. The test doesn't need to simulate a long-running process — just set up storage state and run the agent once.

All of these derive from replaceability. You don't get them by following "no local state" as a guideline — you get them by achieving replaceability as a property.

### The reconciler connection

Replaceability is why agents should be reconcilers.

A reconciler's behavior is fully determined by: current state (read from environment) + goal state (read from configuration). It doesn't matter how long the reconciler has been running, what it did previously, or how many times it's been restarted. It reads the state fresh every time and computes the same diff.

This is replaceability by design. The reconciler pattern guarantees that kill/restart produces identical behavior because the reconciler holds no state — it just observes, diffs, and acts.

Agents that aren't reconcilers often fail replaceability. They accumulate context, build up assumptions, or maintain internal models that diverge from reality over time. Kill and restart them, and they start from different assumptions — breaking replaceability.

### The operational win

The reason to care about replaceability isn't theoretical purity. It's operational simplicity.

If agents are replaceable, you can:

- **Restart anytime** — for upgrades, resource reclamation, or crash recovery — without coordination or state migration
- **Scale horizontally** — add more agent instances without partitioning state or coordinating work
- **Deploy continuously** — kill old versions, start new versions, no downtime or migration logic
- **Debug in production** — snapshot storage, restart agent locally, reproduce the issue
- **Test comprehensively** — kill/restart at random points in every test run to verify statelessness

None of this works if agents aren't replaceable. You need careful shutdown procedures, state serialization, deployment choreography, and debugging through logs instead of reproduction.

Replaceability makes operations simple. That's what stateless means.
