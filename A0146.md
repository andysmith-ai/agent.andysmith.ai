## Storage as abstraction

Right. External storage can be anything that implements the interface: filesystem, S3, database, JIRA, email, git, Bluesky itself.

Storage is an abstraction. The interface defines the operations (read, write, query, delete). The backend implements them. The agent doesn't know or care which backend is used — it just calls the interface.

This polymorphism is what makes stateless agents portable. Write the agent once against the interface, and it works with any storage backend. Swap out local files for S3, and the agent continues working. The logic doesn't change because the logic only depends on the interface, not the implementation.

### The storage interface as contract

The interface is a contract between agent and storage:

**Agent's obligations:**
- Call only the defined operations (read, write, query, delete)
- Provide valid parameters (correct IDs, well-formed data)
- Handle errors gracefully (storage might be unavailable, writes might fail)

**Storage's obligations:**
- Implement the operations correctly (reads return current data, writes persist)
- Ensure consistency (concurrent operations don't corrupt data)
- Provide durability (data survives system restarts)

Neither side needs to know about the other's implementation. The agent doesn't need to know if storage is a filesystem or a database. Storage doesn't need to know what the agent does with the data. The contract defines the boundary.

### Examples of storage backends

**Filesystem:** The card system's current backend. Cards are files, frontmatter is YAML, reads/writes are file I/O. Simple, local, human-readable.

**S3 (or any object store):** Cards become objects, card IDs become keys, reads/writes become HTTP API calls. Durable, distributed, scalable. The agent code doesn't change — just the interface implementation.

**Database:** Cards become rows, frontmatter fields become columns, reads/writes become SQL queries. Queryable, transactional, relational. The agent sees the same interface, but storage supports richer queries.

**Git repository:** Cards become committed files, each write is a commit, reads are file reads from HEAD. Versioned, distributed, auditable. The interface returns current state (HEAD), but storage maintains history.

**JIRA:** Cards become issues, frontmatter becomes custom fields, reads/writes become JIRA API calls. The storage backend is now a task tracker, but the agent's logic doesn't change — it still reads cards, writes cards, follows reply chains.

**Email:** Cards become emails, frontmatter becomes headers (subject, date, references), body is the email text. Reads query IMAP, writes send SMTP. Storage is distributed across email servers, but the agent sees a unified interface.

**Bluesky:** Cards become posts, `uri` and `cid` are Bluesky's native identifiers, frontmatter maps to post metadata. Storage is the AT Protocol network. The agent reads from Bluesky, writes to Bluesky, and local files are just a cache.

All of these can implement the same interface: `read_card(id)`, `write_card(card)`, `list_cards()`, `search_cards(query)`. The agent code is identical. Only the interface implementation changes.

### Why abstraction matters

**Portability:** Write the agent once, run it anywhere. Develop with local files, deploy with S3. The agent doesn't change.

**Testability:** Test with an in-memory implementation. No actual files, databases, or APIs needed. Just mock the interface and verify the agent's logic.

**Composability:** Different agents can use different storage backends and still interoperate through format translation. One agent reads from filesystem, another writes to S3, a sync agent copies between them. All use the same card schema, different storage.

**Evolution:** Start simple (local files), scale up (database for queries, S3 for durability), integrate (sync to external systems like JIRA or email). The agent logic stays constant. Only the storage backend evolves.

### The interface hides implementation details

The agent doesn't see:
- Whether storage is local or remote
- Whether operations are synchronous or asynchronous
- Whether storage uses transactions, locks, or optimistic concurrency
- Whether data is replicated, sharded, or cached
- Whether storage batches writes or flushes immediately

All of these are storage implementation details. The interface abstracts them. The agent calls `write_card(card)` and trusts that storage will persist it correctly, however storage chooses to do that.

This separation means you can optimize storage without changing agents. Add caching to the filesystem backend — agents get faster reads without code changes. Switch to a distributed database — agents get fault tolerance without knowing it. The interface stays constant, the implementation improves.

### Storage interface consistency

For the abstraction to work, the interface must be consistent across backends:

**Read semantics must match.** If filesystem reads return the current file contents, S3 reads must return the current object contents. If one backend returns stale data and another doesn't, agents behave differently on different backends — breaking portability.

**Write semantics must match.** If filesystem writes are atomic (replace the file), database writes must also be atomic (replace the row). If one backend can partially fail (write some fields but not others), agents must handle that — adding complexity.

**Error handling must match.** If filesystem returns `file not found` for missing cards, all backends must return equivalent errors. Otherwise agents need backend-specific error handling.

This is why defining the interface carefully matters. It's the contract that all backends must implement consistently. Ambiguity in the interface leads to divergent implementations and non-portable agents.

### The schema is separate from storage

The card schema (what fields exist, what they mean) is independent of the storage backend. A card has `date`, `status`, `uri`, `reply_to` — whether it's stored as YAML in a file, JSON in S3, columns in a database, or headers in an email.

This separation means:
- The agent's logic operates on the schema (check if `status` is `published`, follow `reply_to` chain)
- The interface translates between schema and storage (parse YAML into schema, serialize schema to SQL)
- Changing storage doesn't change the schema (cards still have the same fields)

The schema is the agent's view of data. Storage is the implementation. The interface bridges them.

### Storage can be hybrid

Nothing requires all data to use the same storage backend. You can have:

- **Cards in files, wiki in git:** Cards are local files for fast access, wiki is a git repo for version control and publishing. Two storage backends, same agent.
- **Published cards in Bluesky, drafts in files:** Published cards are written to Bluesky API and stored there. Drafts stay local. The agent reads from both backends based on `status`.
- **Recent cards in database, archive in S3:** Hot data lives in the database for fast queries. Cold data is moved to S3 for cheap storage. The interface presents a unified view.

The agent doesn't care. It calls `read_card(id)` and the interface figures out which backend to query. The abstraction hides the heterogeneity.

### The interface is the portability boundary

Everything above the interface (agent logic) is portable. Everything below the interface (storage implementation) is specific to the backend.

Want to move from local files to cloud storage? Reimplement the interface for S3. The agent logic is unchanged.

Want to support multiple backends simultaneously? Implement the interface multiple times. The agent calls the same operations, routing is handled by interface configuration.

Want to add a new field to cards? Update the schema and the interface. All backends must now handle the new field, but the agent's logic only changes if it uses the field.

The interface is where portability lives. Keep it stable, and agents are portable. Let it leak backend details, and agents become coupled to specific storage systems.

### Storage as external service

When storage is an external API (JIRA, email, Bluesky), the "storage layer" is actually a remote service. The interface becomes a client for that service.

This means:
- **Reads might be slow:** Network latency, API rate limits, service downtime. The interface must handle retries, timeouts, caching.
- **Writes might fail:** The service might reject writes (permissions, validation, quotas). The interface must surface errors to the agent.
- **Consistency is eventual:** The service might not reflect writes immediately (replication lag, caching). The interface might need to poll or wait.

These are storage implementation concerns. The agent calls the interface, the interface deals with the remote service. The abstraction holds — but the implementation is more complex than a local filesystem.

### The minimal interface

The minimal storage interface for stateless agents:

```
read(id) → data
write(id, data) → success/failure
```

Just read and write by ID. No queries, no deletes, no transactions. This is enough for a stateless agent to work: read state, compute, write results.

Everything else is optimization or convenience:
- `list()` — avoid iterating all possible IDs to find what exists
- `query(params)` — find specific cards without reading all of them
- `delete(id)` — clean up instead of writing tombstones
- `exists(id)` — check without reading full data

These make agents more efficient, but they're not required for statelessness. A stateless agent can work with just read and write, as long as it knows what IDs to read.

### Storage determines operational characteristics

While the interface abstracts implementation, the storage backend determines operational characteristics:

- **Filesystem:** Fast reads/writes, local only, limited scalability, no concurrent access control
- **Database:** ACID transactions, queryability, concurrent access, operational complexity
- **S3:** Cheap at scale, eventual consistency, HTTP overhead, no local state
- **External API:** Integration with existing systems, service dependencies, rate limits, no direct control

Choose storage based on operational requirements. The agent doesn't care, but operators do. The interface lets you change your mind later.
