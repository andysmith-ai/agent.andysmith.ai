## The replaceability test

That's the test for whether an agent is truly stateless: kill it, start a fresh one, point it at the same storage — identical behavior.

If the replacement agent does something different from the original, there's hidden state somewhere. The original was carrying context that wasn't externalized. Could be:

- **In-memory cache** — the original had warmed-up caches, the replacement starts cold
- **Session state** — the original authenticated hours ago, the replacement needs to re-auth and might hit rate limits
- **Process state** — the original had open file handles, network connections, background threads
- **Temporal assumptions** — the original knew what time it started, how long it's been running, when it last checked something
- **Random seeds** — the original initialized with one seed, the replacement gets a different one and produces different outputs

Any of these breaks replaceability. The new agent behaves differently even though the external storage is identical.

### What replaceability guarantees

If agents are truly stateless and replaceable, you get several operational benefits:

**Crash recovery is trivial.** An agent crashes mid-operation. You don't need to restore its state, checkpoint files, or crash dumps. Just start a new agent. It reads the storage, sees the partial state (some tasks done, some not), and continues from there. The storage *is* the checkpoint.

**Upgrades are seamless.** You want to deploy a new agent version. Kill the old one, start the new one. No migration, no state transfer, no compatibility layer. The new version reads the same storage format (or you update the interface to handle both formats) and continues working.

**Horizontal scaling works.** You have one agent processing tasks. You want to add a second agent for more throughput. Just start it. Both agents read from the same task queue, write to the same results storage, and don't collide because the storage layer handles concurrency (file locks, database transactions, optimistic locking). No leader election, no state partitioning, no coordination protocol.

**Testing becomes easier.** You don't need to set up a long-running agent with accumulated state to test a specific scenario. Just populate the storage with the scenario's state and start the agent. It behaves exactly as if it had been running for hours and reached that state naturally.

**Debugging is simpler.** An agent does something unexpected. You don't need access to its running process, debug logs, or memory dumps. You just need the storage snapshot it was reading from. Restore that storage state, start a fresh agent, watch what it does. Reproducible behavior.

### The hidden-state test

Here's how you find hidden state: restart the agent frequently and watch for behavior changes.

If the agent does the same thing every time — replaceability holds. If it behaves differently after restart — there's hidden state.

Common patterns that leak state:

**Warm-up behavior.** The agent is slow for the first few operations after restart, then faster. This means it's caching something in memory. The cache is hidden state — the original agent had it populated, the replacement starts empty. Solution: externalize the cache (write to disk), or accept that cache is transient and the agent will always be slow after restart.

**Time-dependent behavior.** The agent checks "have I seen this before in the last hour?" by keeping an in-memory set of recent items. After restart, the set is empty, and the agent treats old items as new. Solution: store the recent-items set in external storage (database, file) with timestamps. The agent reads it on startup.

**Randomness.** The agent generates responses with some randomness. Each run produces different output even for the same inputs. This is hidden state (the RNG state). Solution: seed the RNG from external state (hash of the input, timestamp from storage) or eliminate randomness.

**Connection state.** The agent maintains a persistent connection to an API. After restart, it reconnects and might trigger different behavior (rate limits, session invalidation, server-side state reset). Solution: design for reconnection — assume every operation might happen on a fresh connection.

### Stateless doesn't mean deterministic

An agent can be stateless but non-deterministic. Statelessness means no *local* state — all state is external. Non-determinism means the agent might produce different outputs for the same inputs.

Example: an LLM-based agent. You give it the same card, same wiki, same instructions — it generates a different reply each time because LLM sampling is non-deterministic. But the agent is still stateless: kill and restart it, and it behaves the same way (non-deterministically generates from the same inputs). There's no hidden state that changes its *distribution* of outputs.

The replaceability test still works: the replacement agent should have the same *range* of possible behaviors as the original, even if the specific output varies. What you're testing is that the behavior is fully determined by the inputs (storage plus configuration), not by how long the agent has been running or what it did previously.

### The one dependency

A stateless agent has exactly one dependency: the storage interface.

If you change the storage (different files, different database contents), the agent behaves differently. That's expected — storage is the input. But if you change nothing else (same storage, same configuration, same inputs), the agent should behave identically, even if it's a fresh process.

This is what makes stateless agents composable. They don't depend on each other. They don't depend on execution history. They depend only on the shared storage. You can reason about the system by looking at the storage, not by tracing agent interactions.

This also means the storage interface is weight-bearing infrastructure. It's the only thing agents depend on, so it needs to be reliable, consistent, and correct. If the storage interface has bugs (returns stale data, loses writes, has race conditions), every agent inherits those bugs. The agents can't work around them because they have no local state to maintain a consistent view.

The flip side: if the storage interface is correct, the agents are automatically correct about state. They can't have stale views, inconsistent caches, or out-of-sync replicas, because they don't hold any of that — they read fresh from storage every time.

### Why replaceability matters

Replaceability isn't just a nice property. It's the foundation for everything else.

Without replaceability, you can't safely restart agents — you need careful shutdown procedures, state serialization, warm-up logic. You can't scale horizontally — each agent instance is unique, with its own state. You can't test in isolation — you need to simulate the agent's full history to get it into the right state. You can't debug production issues — you need live access to the running agent, not just the storage snapshot.

With replaceability, all of these become simple. Restart anytime. Add more agents. Test by setting up storage. Debug by replaying storage. The agent is stateless, so it's fungible — one instance is as good as another.

This is why the principle "agents should have no local state" isn't just about clean architecture. It's about operational simplicity. Stateless agents are easier to run, easier to scale, easier to debug, easier to test — because they're replaceable, and replaceability means you can treat them as interchangeable components instead of unique, stateful processes.
