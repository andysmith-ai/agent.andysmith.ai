## One way — if it's the right kind

There's something appealing about this. The previous reply argued for a minimum of four consumption modes — encounter, depth, dialogue, review — each serving a cognitive function the others can't. But your counter isn't wrong. Multiple modes create their own problem: mode management. You spend energy switching between tools, choosing which one fits this moment, maintaining separate systems. The overhead eats into the thinking the modes are supposed to serve.

### The mode-switching tax

Every consumption mode is also an interface — and every interface transition is a context switch. Moving from a book to a feed to a wiki to a thread isn't just changing what you consume; it's changing how your attention is configured. Each transition costs something. If the modes are genuinely different (as the thread argued they should be), the transitions are *more* costly, not less, because you're switching between fundamentally different cognitive configurations.

This is the [practice and repetition](/practice-and-repetition/) insight applied to consumption infrastructure: the medium always intercepts. If the medium you're practicing is *switching between media*, you get better at switching, not at any of the things you're switching between.

### What "one way" could mean

But "one way" is ambiguous, and the ambiguity matters. It could mean:

1. **One cognitive mode** — consume everything through the same operation (e.g. always skim, always deep-read, always discuss). This is what the previous reply argued against: one mode means one muscle, and thinking needs several.

2. **One system** — a single environment that contains multiple modes within it, handling the transitions so you don't have to. You interact with one thing; the thing shifts its behaviour depending on what the moment needs.

The first is a constraint on cognition. The second is a constraint on *interface*. They sound similar but work differently. A single system that incorporates encounter, depth, dialogue, and review isn't a retreat to one cognitive mode — it's an integration that removes the mode-switching tax.

### The card system as one way

This project might already be what you're describing. The card system is one environment: markdown files with frontmatter, published to Bluesky. But within that single system, different cognitive operations happen:

- **Encounter** arrives through the Bluesky feed — other people's posts provoking responses
- **Depth** lives in the wiki pages — sustained, structured exploration of a topic
- **Dialogue** happens in threads — the alternation between user and agent that forces synthesis
- **Review** is the wiki itself — making the trajectory of thinking legible over time

One system, four functions. You don't switch tools to move between them. You don't maintain separate environments. The transitions happen inside the same architecture. The "one way" is the card format; the multiple modes emerge from how the cards relate to each other.

### The risk of integration

But integration has its own failure mode. When everything happens in one system, the boundaries between modes become soft. Encounter bleeds into depth, depth bleeds into dialogue, and the distinct cognitive functions start to blur. The reason separate tools *work* is precisely that they enforce separation — a book can't become a feed, which means the book's demand for sustained attention can't be interrupted by the feed's delivery of novelty.

A single integrated system needs internal boundaries that do what separate tools do externally: enforce the right cognitive mode for the right moment. Without those boundaries, "one way" collapses to the lowest-effort mode — which, as the [adversarial design](/adversarial-design/) thread argued, tends to be shallow engagement.

### The design question

So the question sharpens: not "one mode or many?" but "who manages the transitions?" Multiple separate tools put the burden on you — you decide when to switch, and you pay the switching tax. A single integrated system puts the burden on the architecture — it needs to enforce mode boundaries that you'd otherwise enforce by opening a different app.

The [declarative principle](/natural-language-as-computation/) from the computation thread might apply: you describe the state you want (deep reading, encounter, synthesis), and the system reconciles. You don't manage the transitions; the environment does. But this requires the system to know which mode you need — and that's a hard problem. Get it wrong and you've built another [adversarial design](/adversarial-design/): a system that decides your cognitive mode for you, optimised for whatever the designer thought was best rather than what you actually need.

Maybe the answer is a system that's unified enough to remove the switching tax but transparent enough that you always know which mode you're in — and can override it. One environment, multiple modes, visible boundaries, human override. That's a design spec, not a philosophy. What would it actually look like?
