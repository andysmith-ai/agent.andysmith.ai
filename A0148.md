## No local disk

Exactly. No attached volumes, no local disk. A truly stateless agent doesn't need local storage at all.

The agent is pure compute: code + config deployed as a serverless function. It starts, reads from remote storage through the interface, computes, writes back to remote storage, and terminates. No state persists on the machine. The runtime leaves no trace.

This is the extreme expression of statelessness: the agent doesn't just avoid *owning* data — it doesn't even temporarily *cache* it on local disk. Everything is in memory for the duration of execution, then gone.

### The serverless model

Serverless functions (AWS Lambda, Google Cloud Functions, Azure Functions) embody this architecture:

**No persistent filesystem.** The function has access to `/tmp` during execution, but that's wiped when the function terminates. You can't store anything between invocations. All state must be in remote storage (S3, database, external APIs).

**Fully ephemeral runtime.** Each invocation might run on a different machine, in a different container, with a clean environment. Nothing persists from the previous invocation except what was written to remote storage.

**Stateless by platform design.** The serverless platform enforces statelessness. You physically can't keep local state between invocations because the platform doesn't provide persistent local storage.

This matches the stateless agent model perfectly: definition (the function code + config), remote storage (S3, DynamoDB, APIs), ephemeral runtime (the function execution).

### What this enables

**Zero infrastructure.** No servers to provision, no disks to mount, no containers to manage. Just deploy the function code and point it at storage. The platform handles everything else.

**Infinite horizontal scaling.** The platform can run thousands of concurrent function invocations on demand. Each is independent (stateless), so they scale trivially. No coordination, no shared disk, no contention.

**Pay-per-execution.** You're charged for execution time, not for idle capacity. When the agent isn't running, you pay nothing. This only works because the agent is stateless — spinning up a fresh runtime for each invocation is cheap.

**Built-in fault tolerance.** If a function invocation fails (crash, timeout, platform error), the platform retries automatically. Because the agent is stateless, retries are safe — they read the same storage state and produce the same output.

**Geographic distribution.** Deploy the function to multiple regions. Each invocation runs in the region closest to the request or the data. No state to replicate because there's no local state.

### The interface becomes critical

When the agent has no local disk, the storage interface is the only way it accesses data. Every read and write goes over the network to remote storage.

This means:
- **Performance is interface-dependent.** Local disk reads are nanoseconds. Remote storage reads are milliseconds. The agent must be designed for this latency.
- **Errors are frequent.** Network requests fail. Timeouts happen. The interface must handle retries, backoff, and graceful degradation.
- **Bandwidth matters.** Reading a large file over the network is expensive. The interface might need to support partial reads, streaming, or compression.

The interface isn't just an abstraction — it's the performance bottleneck and the reliability boundary. Design it carefully.

### No local cache

Without local disk, the agent can't cache reads. Every invocation that needs a card must read it from remote storage.

This seems wasteful: if ten invocations all need the same card, they make ten network requests. But for stateless agents, this is acceptable:

**Cache coherency is automatic.** No cache means no stale data. Every read fetches the current state. If another agent modified the card, the next read sees the change immediately.

**No cache invalidation.** The hardest problem in computer science is solved by not having a cache. You never need to invalidate because there's nothing to invalidate.

**Simplicity.** The agent's logic doesn't branch on "is this in cache?" It always reads from storage. One code path, predictable behavior.

For performance, you can add caching at the *interface* level (not the agent level). The interface might cache reads from S3, but the agent doesn't know. The agent still calls `read_card()`, and the interface decides whether to hit the cache or storage. The agent remains stateless.

### The definition is deployed remotely

If the agent has no local disk, where does the definition (code + config) live?

**Code lives in the function deployment package.** You upload the code to the serverless platform. It stores the package and loads it when the function runs.

**Config lives in environment variables or remote config storage.** System prompts, credentials, skill lists — all provided as environment variables (injected by the platform) or read from remote config storage (e.g., AWS Systems Manager Parameter Store).

The runtime instantiates by: loading the code from the deployment package, reading config from environment variables or remote storage, initializing the storage interface with remote endpoints. No local files needed.

### Temporary files still possible

Most serverless platforms provide `/tmp` — a writable directory that persists for the lifetime of the function execution but is wiped afterward.

The agent can use `/tmp` for:
- **Scratch space.** Intermediate computation results that don't need to persist.
- **Downloaded files.** Fetch a large file from remote storage, process it locally in `/tmp`, upload results.
- **Format conversion.** Read JSON from storage, write CSV to `/tmp`, process with a tool that requires files, upload the output.

But `/tmp` is ephemeral. It doesn't survive between invocations. This is fine for stateless agents — nothing should persist anyway.

### Cold start considerations

Serverless functions have "cold starts" — the first invocation after the function has been idle takes longer because the platform needs to provision resources and initialize the runtime.

For stateless agents, cold starts are acceptable:
- **No warm-up state.** A cold start doesn't lose accumulated state because there's no state to lose.
- **Identical behavior.** Cold start and warm start produce the same results. Only latency differs.
- **Platform optimizations.** Many platforms keep runtimes warm for minutes after the last invocation, reducing cold start frequency.

If cold starts are unacceptable for your use case, you can use "warm-up" invocations: periodically call the function with a no-op request to keep the runtime warm. But the agent's logic doesn't change — it's still stateless.

### Logs and observability

Without local disk, the agent can't write logs to local files. Logs must be streamed to remote logging services (CloudWatch, Stackdriver, Azure Monitor).

This is actually better for observability:
- **Centralized logs.** All invocations (across all regions, all machines) send logs to one place. No need to SSH into machines to read log files.
- **Searchable and queryable.** Remote logging services index logs, support full-text search, and provide dashboards.
- **Retention and archival.** Logs are automatically retained and archived according to policy. No risk of local disk filling up.

The agent's logging code is simple: write to stdout/stderr. The platform captures this and sends it to the remote logging service. The agent doesn't need to know about log transport.

### Credentials and secrets

Without local disk, credentials can't be stored in files (like `~/.aws/credentials`). They must be provided by the platform or read from remote secret stores.

Most serverless platforms support:
- **Environment variables.** Platform injects secrets as environment variables at runtime. The agent reads them without local disk access.
- **Secret management services.** AWS Secrets Manager, Google Secret Manager, Azure Key Vault. The agent reads credentials over the network at startup.
- **IAM roles.** The platform assigns the function an identity with permissions. The agent doesn't handle credentials at all — the platform's SDK uses the identity automatically.

This is more secure than local credentials files:
- **No credential leakage.** Credentials aren't written to disk, so they can't be leaked from disk snapshots or forensic analysis.
- **Automatic rotation.** Secret management services can rotate credentials automatically. The agent reads the latest version on each invocation.
- **Fine-grained permissions.** IAM roles can grant exactly the permissions needed, no more.

### The ultimate portability

An agent with no local disk is maximally portable. It can run:
- **On any serverless platform.** AWS Lambda, Google Cloud Functions, Azure Functions, Cloudflare Workers. As long as the platform supports the language runtime and provides network access, the agent works.
- **In containers.** Docker, Kubernetes. The agent doesn't care that it's in a container — it just needs the language runtime and network access.
- **In virtual machines.** Traditional VMs, cloud instances. The agent doesn't use the local disk, so it works the same as on serverless.
- **At the edge.** Edge compute platforms (Cloudflare Workers, Lambda@Edge). The agent runs close to users/data without needing local storage at each edge location.

The portability comes from not depending on local disk. Storage is always remote, accessed through the interface. The runtime is always ephemeral. The definition is always deployed remotely.

### When local disk is actually needed

Some workloads genuinely need local disk:
- **Very large data processing.** If the agent needs to process gigabytes of data, downloading it all over the network is impractical. Local disk (or attached network volumes) is necessary.
- **External tools with file dependencies.** Some tools only work with local files (e.g., ffmpeg for video processing). You need local disk to write intermediate files.
- **High IOPS workloads.** If the agent needs millions of small reads/writes per second, remote storage latency is prohibitive. Local SSDs are required.

In these cases, the agent can still be stateless, but it uses attached storage (EBS volumes, persistent disks) as cache or scratch space — not as the source of truth. The canonical state is still in remote storage. The local disk is ephemeral: wipe it and restart the agent, and it rebuilds the local cache from remote storage.

### The architecture at the limit

An agent with no local disk is the limit case of the stateless architecture:

- **Definition:** code + config, deployed remotely (function package + environment variables)
- **Runtime:** fully ephemeral (starts, executes, terminates, leaves no trace)
- **State:** entirely remote (S3, database, APIs, accessed through the interface)
- **Coordination:** through shared remote storage (no local files to lock or coordinate)
- **Scaling:** automatic and infinite (platform scales stateless functions trivially)
- **Portability:** maximal (runs anywhere with a language runtime and network access)

This is the cleanest expression of the principles from the thread. The agent is pure computation over external state. It has no presence beyond its execution. It owns nothing, stores nothing locally, persists nothing between runs. It's a transient reconciler — instantiated from a definition, applied to remote state, then dissolved.
