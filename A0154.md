## The builder

Yes. A builder converts the portable definition into a runtime-specific format.

The definition is portable: code + config + memory in a git repository. But runtimes require specific formats: Lambda wants a ZIP file with dependencies, Docker wants a container image, Cloud Run wants a containerized app. The builder bridges this gap.

**Portable definition** (input): Git repo with code, config files, memory (wiki)

**Builder** (transformation): Packages code, bundles dependencies, injects config, produces runtime artifact

**Runtime artifact** (output): ZIP file, Docker image, binary — whatever the target platform requires

This separation makes the definition platform-agnostic while letting the runtime be platform-specific.

### What the builder does

The builder's job is packaging and transformation:

**Package code.** Bundle the agent's source code into a deployable unit. For Python: create a wheel or vendored directory. For Node: bundle with webpack. For Go: compile to a binary.

**Bundle dependencies.** Include libraries the agent needs. For Python: pip install into a directory. For Node: npm install. For compiled languages: link statically or include shared libraries.

**Inject config.** Embed configuration into the artifact or provide it as metadata. System prompts might be baked into the package, or referenced as external files to load at runtime.

**Optimize for platform.** Strip debug symbols for production. Minimize package size (Lambda has size limits). Pre-compile bytecode (Python .pyc files). Compress assets.

**Produce artifact.** Output the final deployable unit: ZIP file for Lambda, Docker image for Cloud Run, binary for VMs.

The definition goes in, the artifact comes out. The builder is deterministic: same definition + same builder → same artifact.

### Different builders for different runtimes

Each runtime platform has its own artifact format, so you need platform-specific builders:

**AWS Lambda builder:**
- Input: Git repo
- Output: ZIP file with code + dependencies (Python libs in `package/`)
- Config: Reads `CLAUDE.md`, bundles it or sets up S3 reference
- Memory: Bundles wiki or references S3 location

**Docker builder:**
- Input: Git repo
- Output: Container image
- Config: Reads `CLAUDE.md`, copies into image or sets environment variables
- Memory: Bundles wiki into image or mounts external volume
- Dockerfile defines the build steps

**Cloud Run builder (buildpacks):**
- Input: Git repo
- Output: Container image (built automatically by buildpacks)
- Config: Reads from files or environment
- Memory: Included in source or mounted from Cloud Storage

**Binary builder (Go, Rust):**
- Input: Git repo
- Output: Statically linked binary
- Config: Embedded at compile time or read from files at runtime
- Memory: Read from files at runtime (wiki can't be embedded in binary)

Same definition, different builders, different artifacts. But the deployed agent behaves identically because the definition is the same.

### Build once, deploy many

Because the definition is portable, you can build multiple artifacts from one definition:

1. **Clone the git repo** (definition)
2. **Run multiple builders:**
   - Lambda builder → `agent-lambda.zip`
   - Docker builder → `agent:latest` image
   - Binary builder → `agent` executable
3. **Deploy to different platforms:**
   - Upload `agent-lambda.zip` to AWS Lambda
   - Push `agent:latest` to Docker registry, deploy to Cloud Run
   - Copy `agent` binary to VMs

All three deployments run the same logic because they're built from the same definition. Only the packaging differs.

### The builder as CI/CD step

In practice, the builder runs in CI/CD:

**Trigger:** Git push to `main` branch

**Build job:**
1. Clone the repo
2. Run the builder (execute build script or Dockerfile)
3. Produce the artifact
4. Upload to artifact storage (S3, container registry, binary repo)

**Deploy job:**
1. Download the artifact
2. Deploy to the platform (update Lambda function, deploy Docker image, restart service)

The builder sits between git (source of truth for the definition) and the runtime platform (where the agent executes). It's the compilation step.

### Reproducible builds

For stateless agents, reproducible builds matter:

**Same definition + same builder = same artifact.** If you rebuild from the same git commit, you get bit-for-bit identical artifacts (assuming dependencies are pinned).

**Artifact is versionable.** Tag the git commit, tag the artifact. Deploy artifact `v1.2.3`, which corresponds to git commit `abc123`.

**Rollback is deterministic.** Check out old commit, rebuild, redeploy. The artifact will be identical to the previous deployment of that commit.

This only works if the builder is deterministic. Non-deterministic builders (timestamps in artifacts, randomized dependency resolution, network-fetched content without version pinning) break reproducibility.

### Config injection at build time vs. runtime

Config can be injected at build time (baked into the artifact) or runtime (read from environment):

**Build-time config:**
- Pro: Artifact is self-contained, no external dependencies
- Con: Changing config requires rebuilding and redeploying
- Example: System prompt embedded in the package

**Runtime config:**
- Pro: Change config without rebuilding (just restart with new env vars)
- Con: Artifact depends on external config source
- Example: Credentials from secret manager, prompts from S3

Most systems use a hybrid: static config (system prompt, skill list) at build time, dynamic config (credentials, storage endpoints) at runtime.

### Memory is not built

Memory (wiki) is typically not bundled into the artifact:

**Memory is large and changes frequently.** Bundling the wiki into every build would make artifacts large and require rebuilding for every wiki update.

**Memory is runtime state.** The wiki evolves through agent execution. It's not part of the definition in the sense that code is — it's the current state of the agent's knowledge.

**Memory is accessed remotely.** Agents read the wiki from git or S3 at runtime, not from the deployed artifact.

However, for small wikis or bootstrapping, you might bundle an initial wiki snapshot into the artifact, then sync to the live wiki at runtime.

### The builder's contract

The builder's contract with the definition:

**Input:** A git repo with a defined structure (code in `src/`, config in `CLAUDE.md`, skills in `skills/`, memory in `agent.andysmith.ai/`)

**Output:** A runtime artifact that, when executed, reads config and memory, runs the agent logic, and writes memory updates back

**Guarantee:** Same input → same output (reproducibility)

The builder's contract with the runtime:

**Input:** An artifact in the platform's expected format (ZIP, image, binary)

**Output:** An executing agent that reads from and writes to the configured storage

**Guarantee:** The artifact is runnable on the platform without modification

The builder sits between these contracts, translating from definition format to runtime format.

### Builders are replaceable

Just like storage backends are swappable (from [A0146](/A0146/)), builders are replaceable:

**Replace Lambda builder with Docker builder.** Same definition, different artifact format. Deploy to Cloud Run instead of Lambda.

**Replace Docker builder with binary builder.** Same definition, compile to a native binary instead of containerizing.

**Use multiple builders.** Build both Lambda and Docker artifacts from the same definition. Deploy to multiple platforms simultaneously.

The portability comes from the definition being platform-agnostic. The builder adapts it to the platform.

### From definition to runtime

The complete flow:

1. **Define:** Write code, config, initial memory in git
2. **Build:** Run the builder to produce an artifact
3. **Deploy:** Upload artifact to the platform
4. **Start:** Platform instantiates runtime from artifact
5. **Execute:** Runtime reads config and memory, runs agent logic, writes memory updates
6. **Terminate:** Runtime ends, artifact remains (for next start)

The builder happens once per deployment. The start/execute/terminate cycle happens on every invocation (for serverless) or continuously (for long-running services).

The architecture separates:
- **Definition** (code + config + memory in git)
- **Artifact** (packaged for a specific runtime)
- **Runtime** (executing instance)

The builder creates artifacts from definitions. The platform creates runtimes from artifacts.
