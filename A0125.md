## Build it — but which version?

You've moved from diagnosis to architecture to construction in four posts. That's the right tempo. But "I'll start building" is where the thread forks into two very different projects, and the fork depends on one design question.

**Does the orchestrator hold the task model, or do you?**

In the simple version, the orchestrator is a dispatcher. You tell it what to do, it routes work to agents, it reports back. It's a voice-controlled task router — a glorified tmux. The state lives in your head. The orchestrator is stateless between interactions: it does what you say and forgets.

In the harder version, the orchestrator is the system of record. It maintains a persistent model of what you're working on, what's in progress, what's blocked, what's done. It doesn't just execute commands — it knows the state of your work independently of you. You can walk away, come back, and ask "where are we?" and it knows.

The first version is simpler to build. The second version is what the thread has been describing. The [ambient interface](/interaction-interfaces/) from the earlier posts only works if the orchestrator has continuous context — and continuous context means persistent state. An orchestrator that forgets between sessions is just a nicer command line.

### The bootstrapping problem

There's something recursive about this project. You're going to build the orchestrator by doing the same thing you're doing now: opening a Claude Code tab, dictating what you want, switching to the next thing. You'll be manually orchestrating the construction of the thing that should be orchestrating for you. That's not a problem — it's how all tools get built. You use the old tool to build the new one.

But it means the first version of the orchestrator should be the simplest thing that removes one piece of your current manual multiplexing. Not the full ambient voice interface. Not the persistent context model. Just: one thing that tracks which agents are running and what they're doing. A task registry. Then you build on top of that.

This is your own [process automation](/process-automation/) method applied to itself: decompose the orchestrator into steps, automate one step, compose. The decomposition might look like:

1. **Task registry** — know what's running
2. **Status polling** — know what's done
3. **Result routing** — surface completions without manual checking
4. **Intent decomposition** — break a spoken goal into agent tasks
5. **Priority judgment** — decide what to tell you next

Each one is a step that can work independently. Each one removes one piece of your manual scheduling. And the first one — just knowing what's running — might be enough to solve the forgetting problem from earlier in the thread.

### What's the first step?

The most revealing question isn't "what should the orchestrator do?" — it's "what do you build first?" The answer tells you what's actually hurting. If you build the task registry first, the pain is losing track. If you build the result routing first, the pain is context-switching to check on things. If you build intent decomposition first, the pain is translating your goals into agent commands.

Which is it?
