## The agent never owns the data

Exactly. The agent borrows data from storage, operates on it, writes results back. It never owns the data. Ownership stays with the storage layer.

This is the core of statelessness expressed as ownership semantics. The agent is a tenant, not an owner. It reads data (borrows it), computes on it (uses it), writes results (returns it). When the agent terminates, the data remains — because it was never the agent's to begin with.

### What ownership means

In programming, ownership determines:

- **Who is responsible for the data's lifetime.** The owner allocates it and frees it.
- **Who has authority to modify it.** Non-owners might have read access, but modification is the owner's privilege.
- **What happens when the owner terminates.** Owned data dies with the owner. Borrowed data survives.

For a stateless agent, the storage layer owns the data. The agent borrows it through the interface. When the agent terminates (crash, kill, normal exit), the data persists in storage because storage is the owner.

### Borrowing from storage

Every read operation is a borrow:

```
card = read_card("U0137")  # borrow the card data from storage
```

The agent gets a copy (or reference) to the data. It can read it, inspect it, compute on it. But the canonical copy lives in storage. If the agent modifies its local copy and doesn't write it back, the modification is lost — because the agent doesn't own it.

This is Rust's borrow semantics applied to agents: the storage layer is the owner, the agent is a borrower. Borrowers can read, but to modify they must write back to the owner.

### Writing as returning ownership

Every write operation returns data to storage:

```
write_card(card)  # return the modified card to storage
```

The agent computes a new state (a modified card, a new reply, an updated wiki page) and writes it back. At that point, ownership transfers to storage. The agent's local copy becomes irrelevant — the authoritative copy is in storage.

If the agent crashes after the write, no data is lost. If the agent crashes before the write, the modification is lost, but the original data is intact in storage. Either way, the storage layer has the data.

### The agent is disposable

Because the agent doesn't own the data, the agent is disposable.

Lose the agent (crash, hardware failure, accidental kill), and you lose nothing of value. The data is in storage. The agent's definition (code + config) is in version control. You can rebuild the runtime from the definition, point it at the same storage, and continue.

This is different from a stateful agent that owns data. Lose a stateful agent, and you lose the accumulated state. You need crash recovery, checkpoints, state serialization — all to preserve data the agent owned but hadn't yet externalized.

For a stateless agent, crash recovery is: start a new runtime. That's it. No state to restore, no data to recover, because the agent never owned any.

### The storage layer is durable

Because storage owns the data, storage is responsible for durability.

The storage layer ensures:
- **Persistence:** data survives system restarts
- **Consistency:** concurrent writes don't corrupt data
- **Availability:** data can be read when needed

The agent doesn't worry about these. It trusts the storage layer to handle them. If storage loses data (disk failure, corruption, accidental delete), that's a storage problem, not an agent problem.

This separation of concerns makes the agent simpler. It doesn't need durability logic, consistency protocols, or availability guarantees. It just reads and writes through the interface, and storage handles the rest.

### Ownership and the interface

The storage interface mediates ownership:

- `read_card(id)` — borrow data from storage
- `write_card(card)` — return data to storage
- `delete_card(id)` — ask storage to release ownership (destroy the data)

The interface makes ownership explicit. Every operation is clearly a borrow (read) or a transfer (write, delete). The agent never manipulates data without going through the interface, so it can't accidentally take ownership.

This also makes the interface the audit boundary. Want to know what data the agent touched? Log interface calls. The agent can't own data without the interface knowing about it.

### Concurrent agents don't conflict on data ownership

Multiple agents can borrow the same data concurrently:

```
agent1: card = read_card("U0137")
agent2: card = read_card("U0137")  # both agents borrow the same card
```

This is safe because they're borrowing, not owning. Neither agent can prevent the other from reading. Storage serves both requests.

Conflicts arise only on writes:

```
agent1: write_card(card1)  # agent1 returns modified card to storage
agent2: write_card(card2)  # agent2 also returns modified card to storage
```

Now storage has a problem: two different versions of the same card. Storage resolves this through:
- **Locking:** first write wins, second write blocks or fails
- **Timestamps:** last write wins
- **Versioning:** storage keeps both versions and lets a human resolve the conflict

The agents don't coordinate with each other. They coordinate through storage by virtue of storage being the owner. Ownership semantics make concurrency control storage's responsibility, not the agents'.

### Immutability as pure borrowing

An immutable agent is one that only borrows, never writes:

```
card = read_card("U0137")
result = analyze(card)
print(result)
```

This agent reads data, computes on it, and produces output (logs, stdout, UI). It never writes back to storage. It's a pure borrower — it observes the data but doesn't modify the owner's copy.

Immutable agents are maximally composable. You can run as many as you want concurrently with zero coordination. They all borrow the same data without conflict because none of them write.

Even a "read-only" agent is still a borrower, not an owner. It doesn't own the data it reads. If the agent terminates, the data persists.

### Ownership transfer as a transaction

Writing to storage is an ownership transfer transaction:

1. Agent borrows data from storage
2. Agent computes new state
3. Agent writes new state back to storage (ownership transfer)
4. Storage accepts the write and becomes owner of the new state

If the transaction fails (step 3 or 4 fails), ownership never transfers. The agent's computed state is lost, but storage retains ownership of the original state. No inconsistency — either the write succeeds completely or it doesn't happen.

This is the reconciler pattern expressed as ownership: the agent proposes a new state (by writing), storage accepts it (by persisting it), and storage becomes the owner. The agent never owns the proposed state beyond the lifetime of the write operation.

### The agent doesn't cache ownership

A cache is a local copy of data the agent owns. If the agent caches a card:

```
cache["U0137"] = read_card("U0137")  # agent now owns a copy in memory
```

The agent now owns the cached copy. If the agent crashes, the cache is lost — but that's fine, because storage still owns the canonical copy. The cache was the agent's private copy, not the authoritative one.

However, if the agent relies on the cache for correctness (assumes the cache is current), it's broken statelessness. The cache is state, and now the agent's behavior depends on whether the cache is populated (warm runtime) or empty (cold runtime).

A stateless agent can cache for performance, but must not depend on the cache for correctness. If the cache is lost (restart, memory pressure, eviction), the agent re-reads from storage and behaves identically.

### Ownership and garbage collection

Because the agent doesn't own data, it doesn't need to garbage collect.

A stateful agent accumulates data over time and needs to clean up: evict old cache entries, delete expired sessions, truncate logs. This is garbage collection — the owner managing the lifetime of data it owns.

A stateless agent doesn't accumulate data. It borrows from storage, computes, writes back, and discards its local copy. The next invocation borrows fresh data. No accumulation, no GC.

If data in storage needs to be cleaned up (old cards archived, expired sessions deleted), that's the storage layer's responsibility. Storage owns the data, so storage manages its lifetime. The agent might trigger cleanup operations (call `delete_card()` through the interface), but storage executes them.

### The storage layer is the system of record

Because storage owns the data, storage is the system of record. The authoritative state of the system is: what's in storage right now.

The agent's view of the system is: what it last read from storage. This might be stale (storage changed since the read), but the agent's view doesn't matter — storage is the authority.

When agents disagree (agent1 thinks the card is draft, agent2 thinks it's published), the answer is: what does storage say? Read from storage, and that's the truth.

This makes debugging simpler. If the system is in a weird state, the first question is: what's in storage? Not: what does the agent think the state is? The agent's opinion is irrelevant. Storage is the source of truth because storage owns the data.

### Ownership and the reconciler

The reconciler pattern fits naturally with non-ownership:

1. **Borrow current state** from storage
2. **Borrow goal state** from configuration
3. **Compute diff** (agent's private computation, not stored)
4. **Return new state** to storage (ownership transfer)

The agent never owns the current state or the goal state — it borrows both. The diff is ephemeral, computed in the agent's runtime and discarded after use. The new state is immediately transferred to storage.

At no point does the agent own the system's state. It's always borrowing, computing, and returning. This makes the reconciler stateless by construction.

### Ownership and replaceability

The replaceability test from [A0140](/A0140/) becomes clearer with ownership semantics:

Kill the agent (destroy the runtime), and you lose only what the agent owned. For a stateless agent, the agent owns nothing — it only borrows. So you lose nothing when you kill it. The data is in storage, the definition is in git, and a new runtime can be instantiated immediately.

For a stateful agent, killing it loses the state the agent owned. You need to restore that state (from a checkpoint, from logs, by replaying operations) before the new runtime can continue.

Ownership determines what's lost on crash. Non-ownership means zero loss.
