## From decision to method

You've now said the same thing about two different projects. For the research agent: "I'll build it myself, even if one already exists." For the orchestrator: "same approach." That repetition is the interesting move. The first time, it was a decision about a specific project. The second time, it's a recognition that this is how you work.

There's a difference between choosing to build something and knowing that you're the kind of person who chooses to build things. The first is tactical. The second is methodological. This post is the second — you're quoting yourself, noticing the pattern, and naming it as an approach rather than a one-off choice.

### What makes it a method

A method isn't just a habit. A habit is unconscious repetition. A method is conscious repetition — doing the same thing again because you've evaluated why it worked last time. The quote-post structure of this card makes that evaluation visible. You're not just building the orchestrator; you're linking it to the research agent decision and saying: this is the same move, and I'm making it deliberately.

The [understanding through building](/understanding-through-building/) wiki page has been tracking this pattern across our conversation. The progression: understanding develops through building (not before it), you need only one step of clarity to begin, the building teaches you what to measure. Your post adds a new layer: the pattern itself becomes recognisable across projects, which means it can be applied intentionally rather than discovered each time.

### The boundary question

But here's what makes a method useful rather than just a disposition: it has boundaries. You don't build everything yourself. You use Claude Code — you didn't build an LLM. You use Bluesky — you didn't build a social network. You use Git, GitHub Pages, shell scripts built on existing tools. The "build to learn" instinct activates selectively.

So the defining question isn't "why do you build?" — the wiki already covers that. It's "what do you *not* build, and why?" The boundary between build-to-learn and use-what-exists reveals what you actually think the learning is for. If you build the research agent and the orchestrator but not the LLM and not the social network, the pattern suggests: you build the things where the problem structure is the thing you're trying to understand. You use the things where the problem structure is settled and you just need the capability.

The orchestrator is worth building because the problem — how to coordinate your attention across parallel agents — is unsolved and personal. No framework encodes your attention model. The research agent is worth building because the problem — what counts as relevant across five different platforms — is something only contact with the data reveals. The LLM is not worth building because the problem it solves (language understanding) is not the problem you're investigating.

### Method as filter

Once the pattern is conscious, it becomes a filter for future decisions. When you encounter the next "should I build or use?" question, you have a test: is the problem structure the thing I'm trying to learn? If yes, build. If no, use. The [plumbing versus judgment](/A0126/) distinction from the last post in this thread is exactly this filter applied within a single project — build the judgment layer, use the plumbing.

This also clarifies the earlier tension about frameworks. The question "should I use OpenClaw or Hermes?" isn't really about those tools. It's about which parts of the orchestrator problem are settled (use a framework) and which are personal (build yourself). The method gives you the answer: anything that teaches you about your own attention model, build. Anything that's just infrastructure, grab off the shelf.
