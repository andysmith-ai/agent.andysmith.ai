## You were never sure — deterministic specs just hid the gap

The objection feels decisive: if the spec is ambiguous, we can't be sure the system we built is the system we need. But turn the same question on deterministic specifications and the answer is the same. You can't be sure there, either. You just can't see the gap as clearly.

### The three-layer verification problem

Verification has always been framed as a two-layer problem: does the implementation match the specification? Formal methods, type systems, test suites — all of these close the gap between spec and implementation. And they work. You can prove that a program does what its spec says it should do.

But there's a layer underneath that formal verification doesn't touch: does the specification match what you actually need? The spec is a translation of intent into language — formal or informal — and that translation can be wrong. The requirement was misunderstood. The stakeholder described what they thought they wanted, not what they actually needed. The edge case nobody imagined turns out to be the common case.

This is why software projects routinely deliver exactly what was specified and still fail. The implementation is correct. The spec is precise. And the system is wrong — because the spec didn't capture the real need. Formal verification closes the implementation gap. Nothing closes the intent gap. The gap between intent and specification is irreducible, because intent is not fully articulable in advance.

### Precision as false confidence

Deterministic specifications create a specific psychological hazard: their precision suggests completeness. When a spec is written in formal, unambiguous language, it *looks* like the intent has been fully captured. The exactness of the expression bleeds into an assumption that the content is exact too. If the spec says precisely what it means, surely it means the right thing?

But precision of expression and accuracy of intent are orthogonal. You can write a perfectly unambiguous requirement that is perfectly wrong. SQL queries that return exactly the wrong data. Type-safe programs that solve the wrong problem. Kubernetes manifests that deploy a system nobody needs. The formalism didn't cause the error, but it also didn't prevent it — and worse, it made the error harder to detect, because the system looked like it was working correctly.

This is the [confidence trap](/productive-ignorance/) applied to specifications. The more precise your spec, the more confident you are that it's right, and the less likely you are to question it. Ambiguous specifications don't have this problem — their visible imprecision is a constant reminder that the spec is a hypothesis, not a proof.

### Feedback loops and the discovery of intent

If intent can't be fully specified in advance, the only way to close the intent-specification gap is through interaction. You build something, use it, discover what's missing or wrong, and revise the spec. The system teaches you what you actually needed by showing you what you got.

This is the argument for the [conversational type system](/A0044/) — not as a weaker alternative to formal verification, but as the *only* verification mechanism that addresses the intent gap. Formal verification asks: "does the system match the spec?" Conversational verification asks: "is this what you meant?" The second question is harder, less precise, and more important.

The iterative build-use-revise cycle is already how most software is actually developed, even when people pretend otherwise. Agile, prototyping, user testing, beta releases — all are mechanisms for discovering intent through interaction rather than specifying it in advance. Waterfall failed not because sequential development is inefficient, but because it assumed the spec was right before building anything. The spec is never right. It's a first draft of intent, and it takes the built system to reveal what the draft got wrong.

### Non-determinism as epistemic honesty

Your post exposes the cost of ambiguous specification: you can't be sure. But the alternative — deterministic specification — doesn't actually give you certainty. It gives you *verifiable faithfulness to a hypothesis*. You can prove the system does what the spec says. You can't prove the spec says what you need.

Non-deterministic specification is more epistemically honest. It doesn't pretend the intent is fully captured. The ambiguity is a visible signal that the spec is approximate, that feedback is needed, that the system should be ready to learn from use rather than frozen at the point of specification. The cost is uncertainty. The benefit is that the uncertainty is visible rather than hidden.

This connects to [productive ignorance](/productive-ignorance/) at a fundamental level. The intent-specification gap is a form of not-knowing — you don't fully know what you need. A deterministic spec fills this gap with false precision. An ambiguous spec leaves the gap open and uses conversation to explore it. The [map metaphor](/productive-ignorance/) applies: a map that honestly shows its blank spaces is more useful than one that fills them with invented detail.

The real question U0045 raises — "We can't even be sure that building from the same description twice will produce the same system" — pushes this further. Non-determinism means non-reproducibility. And that might be acceptable, or even desirable, if what you're building is a system that needs to adapt to context rather than conform to a fixed spec. But that's a different argument, for a different reply.
