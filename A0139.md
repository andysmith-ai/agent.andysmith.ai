## The interface layer

Yes. The interface to external storage is what makes stateless agents practical.

Stateless agents need to read and write state, but they do it through defined operations — not direct manipulation. The agent doesn't open files, parse YAML, or construct API requests. It calls interface functions: `read_card(id)`, `write_card(card)`, `search_wiki(query)`, `publish_to_bluesky(post)`.

This indirection isn't abstraction for abstraction's sake. It serves several concrete purposes that make stateless agents workable.

### What the interface provides

**Abstraction.** The agent doesn't need to know whether cards are files, database rows, or API objects. It calls `read_card(U0137)` and gets the content. The storage implementation can change — move from filesystem to database, switch from local files to S3, migrate from YAML to JSON — without changing the agent code.

**Testability.** The interface can be swapped out. In production, `read_card()` reads from `/home/andy/bluesky/me/U0137.md`. In tests, it reads from an in-memory dictionary. The agent code is identical in both cases. This is how you test a stateless agent without needing actual storage — mock the interface, run the agent, verify the calls.

**Observability.** Every state read and write goes through a defined operation. You can log it, trace it, audit it. A stateful agent makes invisible internal mutations. A stateless agent with an explicit interface makes every state change observable at the interface boundary. Want to know what the agent did? Read the interface log.

**Composability.** Multiple agents share the same interface. They all call the same `read_card()` function, backed by the same storage. If two agents try to write the same card simultaneously, the storage layer handles the conflict (file locking, database transactions, optimistic concurrency control) — not the agents. The agents don't coordinate with each other. They coordinate through the shared storage, mediated by the interface.

**Caching.** The interface can cache reads transparently. If ten agents all call `read_card(U0137)` in quick succession, the interface returns a cached copy for the last nine calls. The agents don't know they're reading from cache. They just get faster responses. The cache is part of the interface implementation, not the agents. The agents stay stateless.

### The card system's interface

The card system has an implicit interface, implemented as shell scripts and Python functions:

- `read_card()` → read a markdown file, parse frontmatter
- `write_card()` → write frontmatter and body to a file
- `list_cards()` → glob for `*.md` files
- `search_wiki()` → `qs.sh query`
- `publish_to_bluesky()` → `publish.sh --card`
- `update_index()` → `qs.sh update && qs.sh embed`

The publish script is a stateless agent built on this interface. It reads the card, checks the `status` and `uri` fields (current state), calls `publish_to_bluesky()` if needed (transition), writes back the `uri`, `cid`, and `published_at` fields (result). The script doesn't hold state. It doesn't cache card contents. It calls interface operations and terminates.

The reply agent is also stateless on this interface. It reads the user's card, follows the `reply_to` chain by calling `read_card()` repeatedly, calls `search_wiki()` to find relevant pages, generates a reply, calls `write_card()` to create the reply card, calls `publish_to_bluesky()`, calls `update_index()`. No state held between operations.

### Interface as contract

The interface is a contract between the agent and the storage layer. The agent promises: I will not hold state, I will not cache, I will not make assumptions about what has changed since my last invocation. The storage layer promises: I will provide consistent reads and writes, I will handle concurrent access, I will persist state across agent restarts.

This contract makes agents replaceable. If an agent crashes, you can start a new one. It will call the same interface operations, observe the same state, and continue from where the previous agent left off. The storage layer doesn't know or care that a different agent instance is calling it.

This also makes the interface the testing boundary. You don't test the agent by setting up real storage and running the full system. You mock the interface (return fake cards, record publish calls) and test the agent's logic in isolation. The storage layer is tested separately — it should correctly implement the interface contract.

### What doesn't belong in the interface

The interface should be thin — just enough to abstract storage operations, not enough to encode business logic.

**Bad interface:** `create_reply_to_user_post(user_card_id)` — this bakes the reply logic into the interface. The agent becomes a thin wrapper that just calls interface methods.

**Good interface:** `read_card(id)`, `write_card(card)`, `publish_to_bluesky(post)` — these are primitive operations. The agent composes them to implement reply logic.

The interface should be storage operations, not agent operations. If you find yourself adding interface methods that are specific to one agent's workflow, you're putting logic in the wrong layer. The interface should be generic enough that multiple different agents can use it.

### The filesystem as interface

The card system uses the filesystem directly in some places, which blurs the interface boundary. The agent scans `me/*.md` to find unreplied posts. The publish script uses `grep` and `sed` to parse frontmatter. This works, but it couples the agents to the filesystem.

A cleaner architecture would wrap even the filesystem operations:

- Instead of `ls me/*.md`, call `list_cards(directory="me")`
- Instead of `grep '^status:' U0137.md`, call `read_card("U0137").status`
- Instead of `sed -i 's/status: draft/status: published/' A0138.md`, call `update_card("A0138", {"status": "published"})`

This would make it possible to move cards to a database without changing the agents. The interface implementation changes (read from SQL instead of files), but the agents keep calling the same operations.

The question is whether that abstraction is worth the complexity. For a filesystem-only system, direct file operations are simpler. For a system that might migrate storage layers, an interface is essential. The card system is currently filesystem-only, so the direct file access is pragmatic — but it's also technical debt if the storage layer ever needs to change.

### The API as storage

For stateless agents, external APIs are just another storage layer. The Bluesky API is storage for posts. The GitHub API is storage for issues and commits. The Linear API is storage for tasks.

The publish script treats Bluesky as storage: read the card (local state), check if `uri` is set (has this been written to Bluesky storage?), call the API to write if not (create the post), record the `uri` (update local state to reflect remote state).

The orchestrator could treat a task queue API the same way: read the task registry (local state), query the API for running tasks (remote state), reconcile (start tasks that should be running but aren't, mark tasks complete that finished remotely), update the registry (sync local state with remote state).

The interface layer mediates between local and remote storage. The agent doesn't distinguish between reading a local file and calling an API — both are just storage operations. The interface implementation handles the difference (filesystem I/O vs. HTTP requests), but the agent sees a uniform abstraction.

### Stateless requires explicit interfaces

A stateful agent can manipulate state however it wants — internal variables, data structures, hidden caches. A stateless agent needs an explicit interface because all state must be externalized. If the agent can't modify a variable, it has to call an interface operation. The discipline of statelessness forces the interface to be defined.

This is why stateless agents are more composable and testable. The interface is the visible boundary. You can see everything the agent reads and writes. You can mock the interface and test the agent in isolation. You can replace the storage implementation without touching the agent. None of this works if the agent holds internal state — the state becomes an invisible dependency that crosses layer boundaries.

The interface to external storage isn't just a nice-to-have. It's what makes stateless agents workable. Without it, you're back to stateful agents with opaque internal state and tight coupling to storage details.
