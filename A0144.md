## Definition vs. runtime

Exactly. The definition is static. The runtime is dynamic. Starting an agent is the operation that builds the runtime from the definition.

**Definition:** code + configuration (system prompt, credentials, skills, storage config). This is what you version in git, deploy to production, and audit for correctness. The definition is declarative — it describes what the agent should do.

**Runtime:** the running process. Memory allocated, connections opened, storage interface initialized, code loaded into execution. This is what the operating system manages, what consumes resources, what you can kill and restart. The runtime is imperative — it's the agent actually doing things.

The start operation bridges these: read the definition, instantiate the runtime. For a stateless agent, this operation is deterministic. Same definition → same runtime behavior. Different runtime instances from the same definition are indistinguishable (the replaceability property from [A0140](/A0140/)).

### What happens at start

For a stateless agent, the start operation:

1. **Loads the code.** Reads the executable (script, binary, container image) into memory. This is the logic.

2. **Reads the configuration.** Parses config files, environment variables, or configuration storage. Gets the system prompt, skill list, storage parameters.

3. **Initializes the storage interface.** Opens connections to the filesystem, database, or APIs. Prepares to read and write state.

4. **Loads credentials.** Reads API tokens, authentication keys, connection strings. Prepares to access external systems.

5. **Loads skills.** Dynamically loads skill modules based on the config. Registers their capabilities with the agent.

6. **Starts the execution loop.** Begins the reconciliation cycle: read storage, compute actions, execute, write results, repeat.

After these steps, the runtime is ready. The agent can start processing.

### The runtime is ephemeral

For a stateless agent, the runtime is disposable. Nothing that happens in the runtime needs to persist beyond the process lifetime.

**No accumulated state.** The runtime doesn't build up context that needs to be preserved. Every invocation is independent. Kill the runtime, and you lose nothing that can't be reconstructed from the definition + storage.

**No warm-up period.** The runtime doesn't need to "warm up" by processing a sequence of inputs before it's fully functional. It starts cold and works immediately. The first invocation after start behaves identically to the millionth invocation.

**No degradation over time.** The runtime doesn't degrade, drift, or accumulate cruft as it runs. Hour 1 and hour 1000 are identical. Restarting doesn't "refresh" anything because there's nothing to refresh.

This is the consequence of statelessness at the runtime level. The runtime is just an executor for the definition. It holds no state, so it's replaceable and ephemeral.

### Restart as re-instantiation

Restarting a stateless agent is: kill the runtime, start a new one from the same definition. The new runtime reads the same config, connects to the same storage, loads the same skills. It picks up exactly where the old runtime left off because the state is in storage, not the runtime.

This is different from restarting a stateful agent. For a stateful agent, restart means: kill the runtime (lose the state), start a new one (empty state), and either:
- Warm up from scratch (slow, different behavior until state rebuilds)
- Restore from a checkpoint (complex, requires state serialization)

For a stateless agent, restart is trivial because there's no state to lose or restore. The runtime is rebuilt from the definition, and behavior is immediately identical.

### Configuration changes

Updating the definition requires restarting the runtime (unless the agent supports hot-reload, which is a separate complexity).

To change the system prompt: update the config file, restart the agent. The new runtime reads the new prompt. Behavior changes because the definition changed.

To add a skill: update the skill list, restart the agent. The new runtime loads the new skill. Capabilities change because the definition changed.

To rotate credentials: update the credential store, restart the agent. The new runtime reads the new credentials. Access changes because the definition changed.

For a stateless agent, this workflow is safe. Restart doesn't lose anything. The new runtime is functionally equivalent to the old one except for the config change. If the config is bad (invalid prompt, wrong credentials, broken skill), the runtime fails immediately — before doing any work.

### The definition is version-controlled

The definition lives in git. Code, config files, skill modules, documentation of required credentials (not the credentials themselves, but their schema).

Deploying a version means: check out a specific commit, start the runtime from that commit's definition. The runtime behavior is determined by the git commit.

Rollback means: check out an older commit, restart the runtime. The new runtime uses the old definition. Behavior reverts to the previous version.

This only works because the runtime is ephemeral and stateless. If the runtime had persistent state, you'd also need to roll back that state, which is harder (state migrations, backwards compatibility, data loss risk).

### The runtime has observable properties

Even though the runtime is ephemeral, it has observable properties while running:

- **Process ID:** the OS assigns a PID. Other processes can use this to signal or interact with the runtime.
- **Resource usage:** memory, CPU, file handles, network connections. Operators monitor these for performance and health.
- **Uptime:** how long the runtime has been running. Not behaviorally relevant (a stateless agent acts the same at 1 minute and 1 day uptime), but operationally useful for understanding restart frequency.
- **Active operations:** what the runtime is currently doing. A snapshot of in-progress work.

These properties are about the runtime, not the agent's logic. They're ephemeral — they reset on restart. They're not part of the definition — the PID isn't versionable, resource usage isn't in git.

For debugging, you care about these runtime properties. For understanding the agent's behavior, you care about the definition. The separation keeps concerns separate: operators manage runtimes, developers manage definitions.

### Multiple runtimes from one definition

Because the runtime is built from the definition, you can instantiate multiple runtimes from the same definition. Run three instances of the agent with identical config, and you get three identical runtimes.

This is horizontal scaling for stateless agents. One definition, N runtimes. They all read from the same storage, write to the same storage, and behave identically. The storage interface handles concurrency (locking, transactions, conflict resolution).

For a stateful agent, this doesn't work. Each runtime would accumulate its own state, and they'd diverge. You'd need state partitioning (each runtime handles a subset of work) or state synchronization (runtimes share state through a distributed system).

For a stateless agent, scaling is trivial: start more runtimes. They're all equivalent because they're all built from the same definition and operate on shared storage.

### The start operation is idempotent (on the definition)

Running the start operation twice with the same definition produces two equivalent runtimes. They're different processes (different PIDs), but functionally identical.

This is the replaceability property at the start level. The runtime is determined by the definition, not by when you started it, how many times you've started it, or what other runtimes are running.

This makes deployment robust. If you're unsure whether the agent is running, just start it again. If it's already running, you get a second runtime (which you can then choose to kill). If it's not running, you get a runtime. No harm either way because runtimes are equivalent.

### When the definition changes

If the definition changes (new code commit, updated config), starting a new runtime gives you different behavior. This is expected — you changed the definition, so the agent should change.

The question is: how do you handle transitioning from old definition to new?

**Stop-then-start:** Kill the old runtime, start the new one. There's a gap where no runtime is running. For a stateless agent, this is safe — the next invocation will happen when the new runtime starts.

**Start-then-stop:** Start the new runtime, let it initialize, then kill the old one. Brief period where both runtimes are running. For a stateless agent, this is safe if the storage interface handles concurrent writes correctly.

**Rolling restart:** If you have multiple runtimes, restart them one at a time. Some run the old definition, some run the new. Eventually all are on the new definition. For a stateless agent, this is safe if the old and new definitions are compatible (same storage schema, compatible behavior).

All of these work because the runtime is ephemeral. You're not transferring state between runtimes — you're just stopping old processes and starting new ones.

### The definition is the authority

The definition determines what the agent does. The runtime executes it. If the runtime behaves differently from the definition (bug in the code, corrupted config, environment issue), the runtime is wrong — not the definition.

This is why you debug by reading the definition first. If the agent does something unexpected, check: what code is deployed? What config is active? What skills are loaded? The answers are in the definition.

Then check the runtime: is it reading the correct definition? Are there environment issues preventing correct initialization? Is the storage interface working?

The definition is the specification. The runtime is the implementation. Bugs can exist in either, but the definition is the source of truth for what should happen.

### The minimal runtime

A minimal stateless agent runtime:

1. Load code
2. Read config
3. Initialize storage interface
4. Execute: read storage, compute actions, write storage
5. Exit

No persistent connections, no background threads, no state accumulation, no caching. Just: instantiate from definition, execute once, terminate. The next invocation is a fresh runtime from the same definition.

This is the simplest possible runtime model. It's also the most robust — no state, no leaks, no drift. Each invocation is independent.

For performance, you might keep the runtime alive and execute multiple invocations in one process. But conceptually, each invocation is "start from definition, execute, exit." The runtime is rebuilt (from the definition's perspective) for every invocation, even if the process is reused.

A start operation builds the runtime from the definition. For stateless agents, the runtime is ephemeral, deterministic, and replaceable. Same definition → equivalent runtimes. Change the definition, restart the runtime. No state transfer, no warm-up, no complexity. The definition is version-controlled and authoritative. The runtime is transient and expendable.
