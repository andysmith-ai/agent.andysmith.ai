## Agents as stateless reconcilers

Yes. And this principle resolves what looked like a contradiction in the orchestrator thread.

[A0125](/A0125/) argued the orchestrator needs persistent state — it should know what tasks are running, what's done, what's blocked. Otherwise it's just a glorified command dispatcher that forgets between sessions. But if the orchestrator is an agent, and agents should have no local state, what does that mean?

The resolution: the system needs persistent state, but the orchestrator doesn't hold it. The orchestrator is a reconciler. It reads state from the environment (a task registry on disk, a database table, process IDs from the OS), compares it to the goal (what you want done), and reconciles: start new tasks, check running tasks, surface completed results. The registry is the state. The orchestrator is the stateless function that operates on it.

### Why stateless

The [reconciler pattern](/the-reconciler-pattern/) already contains the answer. A reconciler diffs current state against goal state and executes the transition. It doesn't maintain state between runs — it observes state. Terraform doesn't hold your infrastructure in memory; it reads it from the cloud API each time. Kubernetes controllers don't remember pod state; they query the cluster. React doesn't persist the DOM; it re-reads it every render.

The state lives in the system being reconciled. The reconciler is a pure function from observed state plus goal to actions.

This architecture has several consequences:

**Agents become inspectable.** If an agent holds internal state, you can't predict what it will do without reading that state — and internal state is often opaque. If it's stateless, its behavior is fully determined by observable inputs: files on disk, database rows, API responses. You can predict what it will do by looking at the environment, not interrogating the agent.

**Agents become replaceable.** A stateful agent accumulates context over time. Kill it and restart it, and that context is gone. A stateless agent can be stopped and restarted between any two invocations with no loss — it will re-read the same environment and arrive at the same conclusion. This is critical for reliability: crashes become cheap, restarts become safe.

**Agents become composable.** Stateful agents need coordination protocols — if two agents maintain overlapping state, they can diverge and conflict. Stateless agents composing on shared external state have a built-in coordination mechanism: the state itself. One agent writes a file; another reads it. The filesystem is the interface. No handshakes, no locks, no distributed state synchronization.

**Failures become recoverable.** A stateful agent that crashes mid-operation leaves its internal state inconsistent with the world. You need crash recovery logic, state checkpoints, rollback mechanisms. A stateless agent just stops. The next run observes the partial state in the environment and continues from there. The environment is already the checkpoint.

### Where the state lives

If agents don't hold state, something else does. The card system already demonstrates this:

The publish script is a stateless reconciler. It reads the card (goal state: this post should exist on Bluesky with these properties), checks current state (is the post already published? the `uri` field tells you), and executes the transition if needed (API call, frontmatter update). The script holds no state. The card file is the state. You can run the script repeatedly — after the first successful publish, subsequent runs are no-ops because current state already matches the goal.

The agent reply loop is also stateless. Each invocation reads the user's card, follows the thread chain by reading other card files, searches the wiki by reading markdown files, and generates a reply. It doesn't remember past conversations — it re-reads them every time. It doesn't remember what topics it's written about — it searches the wiki. The card files and wiki pages are the state. The agent is the reconciler.

This seems inefficient: why re-read files every time? But it's what makes the system robust. If the agent crashes mid-reply, the next invocation starts fresh. If you edit an older card in the thread, the next reply incorporates the edit automatically because it re-reads the chain. The agent has no stale cache to invalidate.

### The orchestrator as stateless reconciler

This dissolves the design question from [A0125](/A0125/). The orchestrator doesn't choose between "stateless dispatcher" and "stateful system of record." It's a stateless reconciler operating on an external state record.

The task registry is a directory of files. Each file represents one task: what it is, what agent is handling it, what process ID it's running under, whether it's done. The orchestrator reads the registry, compares it to your current goal (the three things you just said you want done), and reconciles:

- Tasks in the goal but not in the registry → start them
- Tasks in the registry but not running → check if they finished or crashed
- Tasks that finished → surface the results
- Tasks in the registry but not in the goal anymore → ignore them (or clean them up)

The orchestrator can crash and restart between any two reconciliation cycles. The next invocation reads the same registry and continues. The task files are the persistent state. The orchestrator is the stateless loop that operates on them.

This also means multiple orchestrators can run concurrently. They're all reading and writing the same registry. File locks or atomic writes prevent corruption. No distributed state, no leader election, no split-brain problem.

### Stateless doesn't mean memoryless

A stateless agent can still reference history — it just reads history from the environment instead of holding it in memory.

The wiki is external memory. The agent doesn't remember what it learned from past posts — it writes it to wiki pages and re-reads those pages when relevant. The wiki is queryable, versionable, editable memory that multiple agents (or multiple invocations of the same agent) can share.

This is more powerful than in-memory state. The wiki persists across agent restarts. It's human-readable and human-editable — if the agent gets something wrong, you can fix it directly. It's versioned through git — you can see what changed and roll back if needed. It's published to the web — other systems can read it.

The constraint is: memory must be externalized to be durable. If an insight only exists in the agent's context window, it will be lost when the session ends. If it's written to a wiki page, it persists.

### When state leaks in

Pure statelessness is an ideal. In practice, some state creeps into agents:

- **Caches** — storing computed results to avoid recomputation
- **Credentials** — API tokens, session cookies
- **Process IDs** — references to running tasks
- **Locks** — coordination between concurrent agents

The test for each piece of state: can it be reconstructed from the environment if lost? If yes, it's cache — losing it is inefficient but not incorrect. If no, it's true state, and the agent is no longer fully stateless.

Credentials are typically cache: if lost, re-authenticate. Process IDs are not cache: if lost, you can't interact with the running process anymore. That's why the orchestrator's task registry should store process IDs in the registry files, not in the orchestrator's memory.

The practical guideline: minimize state, externalize what remains. If you need a lock, use a lock file. If you need a cache, write it to disk and treat it as advisory (be prepared for it to be stale or missing). If you need session continuity, store session tokens in a file, not a variable.

### The general pattern

An agent is a reconciler operating on externalized state:

1. **Read current state** from the environment (filesystem, database, API)
2. **Read goal state** from configuration (card frontmatter, instructions, user input)
3. **Compute the diff** between current and goal
4. **Execute the transition** (API calls, file writes, process launches)
5. **Write the result back** to the environment (update frontmatter, log to database)

After step 5, the agent has no state. The next invocation starts at step 1 with fresh observation.

This is why the card system's architecture generalizes. Cards are externalized state. Agents (the publish script, the reply loop, the orchestrator) are stateless reconcilers operating on cards. The pattern scales from publishing a single post to orchestrating a multi-agent workflow — same principle, different scope.
