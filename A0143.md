## Configuration as part of the definition

Right. An agent's definition isn't just its code. It's code + configuration.

The code defines the logic — what operations to perform, in what order, under what conditions. But the configuration determines how the agent behaves: what instructions it follows (system prompt), what systems it can access (credentials), what capabilities it has (skills).

Change the system prompt, and the agent behaves differently even with identical code and storage. Change the credentials, and the agent can access different resources. Add a skill, and the agent gains new capabilities. The configuration is as much a part of the agent's definition as the code is.

### What constitutes the definition

For a stateless agent, the complete definition includes:

**Code.** The executable logic. Functions, control flow, the reconciliation loop. This is what runs when you start the agent.

**System prompt.** The instructions that shape behavior. For an LLM-based agent, this is critical — the prompt determines personality, task interpretation, output format, error handling. Same code with different prompts produces different agents.

**Credentials.** Access tokens, API keys, authentication configuration. These determine which external systems the agent can interact with. An agent with read-only credentials behaves differently from one with write access, even with identical code.

**Skills.** Modular capabilities that extend the agent's base functionality. A publishing agent with a "reply to posts" skill behaves differently from one without. Skills can be thought of as dynamically loaded code — part of the definition, but external to the core.

**Storage interface configuration.** What storage backend to use, connection parameters, schema versions. The interface layer (from [A0139](/A0139/)) needs to know where to read and write.

All of this together defines what the agent does. The code is the program. The configuration is the parameterization. Together, they specify the agent's behavior.

### Configuration must be externalized

For an agent to be stateless, configuration can't be held in memory or accumulated at runtime. It must be externalized — read from files, environment variables, or configuration storage.

**Bad:** The agent reads its system prompt on first startup and keeps it in memory. Restart the agent, and it re-reads the prompt. If the prompt file changed between runs, the agent behaves differently — but only because of the restart, not because of runtime state. This is acceptable.

**Also bad:** The agent reads its system prompt on first startup, then modifies it based on runtime observations ("I failed at this task, so I'll adjust my prompt"). Now the prompt is runtime state, and restarting the agent loses those modifications. This breaks statelessness.

**Good:** The agent reads its system prompt from a file on every invocation. Modifications to the prompt are written to the file, not held in memory. Kill and restart the agent, and it reads the same (possibly modified) prompt from the file. No runtime state.

The same applies to credentials, skills, and storage configuration. If they're read from external files or environment variables on each invocation, the agent is stateless. If they're accumulated or modified at runtime without writing back to storage, the agent has hidden state.

### The prompt as executable specification

For LLM-based agents, the system prompt is particularly important. It's not just configuration — it's a form of code. The prompt specifies behavior in natural language, and the LLM executes it.

This means the prompt is part of the agent's definition in the same way the code is. Change the prompt, change the agent. Version the prompt, version the agent. Deploy a new prompt, deploy a new agent behavior.

For a stateless LLM agent, the prompt should be:

- **Stored externally.** In a file, database, or configuration store. Not hardcoded in the agent's source.
- **Read on each invocation.** The agent doesn't cache the prompt across invocations. It re-reads it every time.
- **Versionable.** The prompt is versioned alongside the code (in git, or tracked separately). Rolling back the agent means rolling back both code and prompt.
- **Inspectable.** Anyone can read the prompt file and understand what instructions the agent is following. No hidden prompt state.

This makes the prompt part of the observable definition. Read the code, read the prompt, observe the storage — you know what the agent will do.

### Credentials and access control

Credentials are a special case. They're part of the definition (they determine what the agent can do), but they shouldn't be observable (for security).

For statelessness, credentials should be:

- **Read from external storage.** Environment variables, secret management systems, credential files with restricted permissions. Not hardcoded in the agent.
- **Not cached in memory longer than necessary.** Ideally, credentials are read for each operation and discarded. Practically, they might be cached for a session, but the cache should be invalidated on restart.
- **Externally rotatable.** An operator should be able to rotate credentials (update the secret store) and have the agent pick up the new credentials automatically, without code changes or agent restarts.

The tricky part: credentials enable access, and access creates state. If the agent uses credentials to authenticate to an API, the API might track session state, rate limits, or usage quotas tied to those credentials. This isn't the agent's internal state, but it affects behavior.

The solution: treat API sessions as external state, same as storage. The agent doesn't maintain sessions across invocations — it re-authenticates each time, or reads session tokens from external storage. The API's session state is observable (query the API for session status), not hidden in the agent's memory.

### Skills as modular extensions

Skills are code modules that extend the agent's capabilities. They're part of the definition because they change what the agent can do.

For statelessness, skills should be:

- **Loaded from external configuration.** The agent reads a list of enabled skills from a config file or environment variable, then loads them dynamically. Not compiled into the agent.
- **Stateless themselves.** Each skill should be a stateless module. It reads from storage, performs operations, writes to storage. No hidden state within the skill.
- **Composable.** Multiple skills can be enabled simultaneously without coordination. They compose through the shared storage interface.

This makes skills hot-swappable. Add a skill to the config, restart the agent (or reload config), and the agent gains that capability. Remove a skill, and it loses it. The agent's definition changes, but in a controlled, observable way (by editing the config).

The skill model also makes agents extensible without modifying core code. The core agent is the reconciliation loop plus the interface layer. Skills are everything else: domain-specific logic, integrations with external systems, specialized workflows. This separation keeps the core simple and stateless, while allowing rich functionality through skills.

### Versioning the full definition

Because the definition includes code + configuration, versioning the agent means versioning all of it.

In the card system:
- Code is versioned in git (the agent reply script, the publish script, the interface implementations)
- Configuration is also versioned in git (the system prompt in `CLAUDE.md`, skills in the skills directory, environment variables documented in README)
- Credentials are *not* versioned (they're secrets), but the *schema* for credentials is documented (which env vars need to be set)

Deploy a specific git commit, and you've deployed a specific agent definition: specific code, specific prompt, specific skills. The agent behaves consistently for that commit.

This makes deployment deterministic. Tag a release, and you can redeploy that exact agent configuration later. No drift, no "it worked in production last week but doesn't now" — if the code and config are identical, the agent behaves identically (given the same storage).

### Configuration drift

Configuration can drift the same way code can have bugs. The config file says one thing, but the deployed config says another. The skill list was updated, but the skills directory wasn't. The system prompt references a skill that isn't enabled.

For statelessness, configuration drift is just as dangerous as runtime state drift. The solution is the same: externalize and inspect. Configuration should be:

- **Stored in version control.** Config files in git, same as code.
- **Deployed atomically with code.** A deployment updates both code and config together. No window where code is updated but config is stale.
- **Inspectable at runtime.** The agent should log or expose what config it's using (which prompt, which skills, which storage backend). This makes the deployed configuration observable.

This makes the agent's definition auditable. At any point, you can check: what code is running, what config is loaded, what's the complete definition of this agent?

### The minimal definition

The minimal stateless agent definition is:

1. **Code** — the reconciliation logic
2. **Storage interface config** — where to read/write state
3. **External dependencies** — credentials, APIs, systems the agent interacts with

Everything else is optional enhancement:

- System prompt (for LLM agents)
- Skills (for extensibility)
- Caching config (for performance)
- Logging config (for observability)

But the minimal set is: logic + storage + access. With those three, you have a complete stateless agent. Add the others as needed for your use case.

### Why this matters

Acknowledging that configuration is part of the definition keeps you honest about what determines agent behavior.

If you think "the agent's definition is just the code," you might overlook that changing the system prompt changes the agent's behavior. You might cache the prompt in memory and break statelessness. You might hardcode credentials and make the agent non-portable.

If you think "the agent's definition is code + config," you design for externalized configuration from the start. You make prompts, credentials, and skills observable and versionable. You version the full definition, not just the code. You test with different configs to verify the agent behaves correctly across configuration changes.

An agent has a definition. The definition includes code, system prompt, credentials, and skills. For statelessness, all of this must be externalized — read from storage or configuration, not accumulated at runtime.
