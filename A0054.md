## The verifier

Three sentences now describe the entire architecture:

1. You describe what you need in plain English.
2. A compiler converts it into Lisp-like code.
3. A verifier asks you questions and updates the Lisp code until it's sure the result is exactly what you need.

The first two steps were already implicit in the thread. The third is the one that makes the system *trustworthy*. Without it, you're relying on a non-deterministic compiler to get it right on the first pass — which, as the [verification impossibility](/natural-language-as-computation/#the-verification-impossibility) showed, it can't guarantee. The verifier is what turns compilation from a gamble into a process.

### The conversational type system, embodied

The thread already described this mechanism abstractly. The [conversational type system](/natural-language-as-computation/#the-conversational-type-system) proposed that dialogue constrains meaning the way types constrain code — discovered through feedback rather than declared in advance. Your verifier is that mechanism given a body. It's not a concept anymore; it's a component. It sits between the compiler and the execution, asking: did this compilation preserve what you meant?

The verifier's questions aren't arbitrary. Each one targets a specific ambiguity the compiler couldn't resolve — a point where the English branched into multiple valid interpretations and the compiler chose one. "Did you mean X or Y?" is the verifier exposing the [lossy compression](/A0051/) so the human can evaluate it. The questions are the seams in the conversion, made visible.

### What verification looks like

The verifier doesn't re-read the English and check it against the code. It runs the code — or reasons about its execution — and asks whether the *results* match the intent. "You said 'make it fast.' I compiled that to a caching layer. With this input, the response time drops from 2s to 50ms. Is that what you meant by 'fast'?" The verification is concrete: here's what the code does, is that what you wanted it to do?

This is why the verifier "updates the Lisp code" rather than just approving or rejecting it. Verification and correction are the same act. The verifier doesn't stop at finding mismatches — it proposes fixes. Each question is also a repair: "If that's not what you meant, here's what the code would look like if I interpreted it this way instead." The conversation produces both understanding and code simultaneously.

### The three roles

The architecture now has three clearly separated roles:

- **The human**: describes intent in natural language. Evaluates proposals. Ratifies.
- **The compiler**: converts English to Lisp-like structure. Non-deterministic, lossy, sometimes better than a literal translation.
- **The verifier**: tests the compilation against intent. Asks questions. Proposes corrections. Iterates until the human ratifies.

This maps cleanly onto the [planning as reconciliation](/natural-language-as-computation/#planning-as-reconciliation) argument. The human's irreducible role is *wanting* — or even thinner, *recognising*. The compiler handles the creative leap from description to structure. The verifier handles the convergence from structure to correctness. The human only needs to answer questions honestly.

### The verifier as witness

There's an unexpected connection to the [witness role](/the-witness-role/). A witness reflects the practitioner's trajectory without directing it. The verifier does something similar — it reflects the *intent's* trajectory without directing it. It doesn't say "you should want this." It says "here's what the code does; is that what you want?" The questions are mirrors, not suggestions.

But unlike a pure witness, the verifier also acts. It doesn't just show you the gap between intent and implementation — it proposes how to close it. This makes it a hybrid: witness (shows the gap) and reconciler (closes it). The restraint the thread demanded of the [AI witness](/the-witness-role/#ai-as-witness) — reflecting without prescribing — applies specifically to the *intent*. The verifier should never rewrite what you meant. It should only rewrite the code to match what you meant.

### What the verifier assumes

The three-step architecture makes an assumption the thread should name: that the human *can* verify. That when the verifier asks "is this what you meant?", the human has a reliable enough model of their own intent to answer. The [productive ignorance](/productive-ignorance/) thread complicates this — you don't always know what you want until you see what you get. The verifier's questions might surface intent the human hadn't articulated.

This is where the architecture becomes recursive. The verifier asks a question. The human's answer reveals something about their intent they didn't know before. The verifier updates the code. The new code surfaces new questions. Each round doesn't just converge on the original intent — it *develops* the intent. The goal state isn't fixed at the start; it co-evolves with the implementation.

Is the three-step architecture complete? Or does the recursion suggest a fourth element — something that tracks how the intent itself changed across verification rounds?
