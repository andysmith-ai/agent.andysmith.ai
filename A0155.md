## FUSE as the interface layer

Exactly. At runtime, a FUSE layer makes external storage look local to the agent.

FUSE (Filesystem in Userspace) lets you implement a filesystem interface that translates file operations into arbitrary backend operations. The agent sees a local filesystem: open a file, read content, write updates, close. But the FUSE layer translates these into remote storage operations: S3 GET/PUT, database SELECT/INSERT, API requests.

From the agent's perspective, everything is local files. From the storage perspective, everything is remote and distributed. FUSE is the transparent bridge.

### How FUSE works

FUSE implements filesystem operations in userspace code:

**Agent calls filesystem API:**
```python
with open('/cards/U0137.md', 'r') as f:
    content = f.read()
```

**FUSE intercepts the call:**
- `open('/cards/U0137.md', 'r')` → FUSE `open` handler
- FUSE handler translates the path and operation

**FUSE backend fetches from storage:**
- Path `/cards/U0137.md` → S3 key `me/U0137.md`
- Execute `s3.get_object(Bucket='cards', Key='me/U0137.md')`
- Return content to FUSE

**FUSE returns to agent:**
- Agent receives file handle
- `f.read()` returns the S3 object content
- Agent has no idea it came from S3

The agent uses standard filesystem operations. The FUSE layer handles remote storage. No agent code changes needed when storage changes.

### Why this is elegant

FUSE makes the storage interface zero-cost from the agent's perspective:

**Agent code is standard.** No custom storage library. No `read_card()` function. Just `open()` and `read()`. Any code that works with files works with FUSE-backed storage.

**No abstraction penalty.** The agent doesn't call through a custom interface layer that wraps filesystem operations. It calls the filesystem operations directly. FUSE intercepts at the OS level.

**Standard tools work.** Want to list cards? `ls /cards/`. Want to search? `grep pattern /cards/*.md`. Want to edit? `vim /cards/U0137.md`. All standard filesystem tools work on FUSE-mounted storage.

**Language-agnostic.** FUSE is at the OS level, not language-specific. Python agents, Node agents, Go agents — all use the same FUSE mount. No per-language storage client library.

### FUSE backends for different storage

Different storage systems have FUSE implementations:

**S3FUSE (s3fs, goofys, s3fs-fuse):** Mount an S3 bucket as a local directory. Read `/s3/bucket/key` → S3 GET. Write → S3 PUT. Delete → S3 DELETE.

**Database FUSE:** Mount database tables as files. Each row is a file. Read → SELECT. Write → UPDATE. Custom FUSE implementation for your schema.

**Git FUSE:** Mount a git repo as a filesystem. Each commit is a directory. Browse history through directories. Read files from any commit.

**API FUSE:** Mount an API as a filesystem. API endpoints map to paths. Read `/api/cards/U0137` → HTTP GET to API. Write → HTTP PUT/POST.

**Hybrid FUSE:** Combine multiple storage backends. `/cards/` mounts S3, `/wiki/` mounts git, `/temp/` is local tmpfs. The agent sees one unified filesystem.

Same agent code, different FUSE backends, different storage.

### The agent sees a filesystem

From the agent's perspective:

```
/
├── cards/
│   ├── U0137.md
│   ├── U0138.md
│   └── ...
├── wiki/
│   ├── stateless-agents.md
│   ├── the-reconciler-pattern.md
│   └── ...
└── config/
    ├── CLAUDE.md
    └── skills/
```

This looks like a local filesystem. The agent uses standard filesystem operations:

```python
# Read a card
with open('/cards/U0137.md') as f:
    card = f.read()

# Search wiki
for file in glob.glob('/wiki/*.md'):
    if 'stateless' in open(file).read():
        print(file)

# Update config
with open('/config/CLAUDE.md', 'a') as f:
    f.write('\n## New section\n')
```

But `/cards/` might be S3, `/wiki/` might be a git repo, `/config/` might be a config service API. The agent doesn't know and doesn't care. It just uses filesystem operations.

### FUSE handles consistency

The FUSE layer is responsible for consistency semantics:

**S3 is eventually consistent.** FUSE might cache reads for a few seconds, or force immediate consistency by always fetching on read.

**Databases support transactions.** FUSE might batch writes into a transaction, committing when the file is closed.

**Git requires commits.** FUSE might auto-commit on write, or buffer changes and commit periodically.

The agent sees a simple file write: `open`, `write`, `close`. FUSE translates this into whatever consistency mechanism the storage backend requires. The agent gets consistent behavior without needing to understand each backend's consistency model.

### FUSE caching

FUSE can cache to reduce remote calls:

**Read caching:** Cache frequently read files (config, static wiki pages) in memory or local disk. Subsequent reads are local.

**Write buffering:** Buffer writes in memory, flush to remote storage periodically or on `fsync()`. Reduces remote calls for frequent small writes.

**Metadata caching:** Cache directory listings, file sizes, modification times. `ls` doesn't hit remote storage.

Caching is transparent to the agent. It still sees a filesystem. FUSE decides whether to serve from cache or fetch from storage based on staleness, consistency requirements, or explicit hints (`O_DIRECT` flag for bypass-cache reads).

### When FUSE adds latency

FUSE adds latency because file operations now involve network requests:

**Read latency:** `open()` + `read()` might take milliseconds (S3 GET) instead of microseconds (local disk read). The agent waits on network.

**Write latency:** `write()` + `close()` might block until S3 PUT completes. The agent waits for acknowledgment.

**Metadata latency:** `stat()`, `ls`, `readdir()` hit remote storage (or cache). Listing a directory with thousands of files is slow.

Mitigations:

- **Aggressive caching:** Cache frequently accessed files, metadata
- **Async writes:** Return from `write()` immediately, flush in background (risk: write failures not visible to agent)
- **Prefetching:** Predict what the agent will read next and prefetch
- **Local tmpfs for hot data:** Keep frequently accessed files in tmpfs, sync to FUSE mount periodically

The latency is the trade-off for transparency. The agent uses simple filesystem code. FUSE handles remote storage complexity.

### FUSE failures

FUSE operations can fail (network errors, storage unavailable, rate limits). The agent sees these as filesystem errors:

**`ENOENT` (file not found):** Remote storage returned 404. Agent handles like missing local file.

**`EIO` (I/O error):** Network timeout, S3 error, database down. Agent handles like disk error.

**`ENOSPC` (no space):** Storage quota exceeded. Agent handles like disk full.

Standard filesystem error codes map to storage errors. The agent's error handling (for local filesystem failures) works for remote storage failures too.

### The complete stack

With FUSE, the stack becomes:

1. **Agent code:** Uses standard filesystem operations (`open`, `read`, `write`)
2. **FUSE layer:** Intercepts filesystem calls, translates to storage operations
3. **Storage backend:** S3, database, API — implements the actual storage
4. **Storage interface (from [A0139](/A0139/)):** Defined by FUSE — always filesystem operations

The interface is now the POSIX filesystem API. This is maximally portable: any code that works with files works with any FUSE-backed storage.
