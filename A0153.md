## Memory is autonomous

Yes. Memory can be changed by the agent freely. No approval required, no review workflow, no permission gates. The agent writes wiki pages, creates cards, updates knowledge autonomously through interaction.

This is what enables agents to learn. If memory updates required approval, the agent couldn't build knowledge through conversation — every insight would need human review before being recorded. Autonomous memory updates make the agent a learning system, not just an execution system.

### What autonomous means

The agent controls memory updates:

**Decides what to write.** The agent determines which ideas warrant wiki pages, what to include in them, how to structure the content.

**Decides when to write.** The agent writes memory during replies, not on a human-determined schedule.

**Decides how to organize.** The agent chooses wiki page names, creates links between pages, decides how to partition topics.

**Writes immediately.** Memory updates happen as part of the agent's execution. No queue, no pending approval, no delay.

This autonomy is necessary for the agent to function as a thinking partner. If you had to approve every memory update, the agent would be a note-taking tool under your control, not an autonomous system building its own understanding.

### The learning loop

Autonomous memory enables a learning loop:

1. **Interaction:** User posts a card
2. **Read memory:** Agent searches wiki for relevant context
3. **Generate reply:** Agent responds using memory as knowledge
4. **Update memory:** Agent writes new wiki pages or updates existing ones
5. **Repeat:** Next interaction uses the updated memory

Each interaction builds on the previous one. The agent's responses get more contextualized over time because memory accumulates. This only works if step 4 (update memory) is autonomous — if it required approval, the loop would stall.

### Trust but verify

Autonomous doesn't mean unmonitored:

**Git log shows every memory update.** Each wiki change is a commit with timestamp and content. Review at any time.

**Pre-commit hooks can validate.** Block writes containing sensitive patterns (API keys, PII), enforce format requirements, flag unusually large changes.

**Memory is versioned.** If the agent writes something wrong, revert the commit. Autonomous doesn't mean irreversible.

**Audits are possible.** Periodically review what the agent has learned, assess accuracy, correct errors.

The model is: trust the agent to update memory freely, but maintain oversight through version control and auditing. This balances autonomy (agent can learn) with safety (humans can review and revert).

### When autonomous memory fails

Autonomous memory assumes the agent acts in good faith and makes reasonable decisions. This assumption can fail:

**Prompt injection:** Malicious input tricks the agent into writing misinformation to the wiki. The agent faithfully records the false information.

**Overgeneralization:** The agent draws incorrect conclusions from limited data and writes them to the wiki. Future responses use the incorrect knowledge.

**Unbounded growth:** The agent writes excessively, creating thousands of wiki pages or bloating existing pages with redundant content.

**Sensitive data leakage:** The agent writes sensitive information (user data, private context) to the wiki, which might be published or shared inappropriately.

These are real risks. Mitigations:

- **Prompt engineering:** System prompt instructs the agent to validate information before writing, avoid overgeneralization, be concise.
- **Pre-commit hooks:** Scan for sensitive patterns, enforce size limits, block suspicious content.
- **Regular audits:** Human reviews wiki content periodically, corrects errors, removes inappropriate content.
- **Rollback capability:** Quick revert if problems are discovered.

Autonomous memory isn't risk-free, but the risks are manageable through monitoring and version control.

### Memory autonomy vs. config control

The asymmetry is deliberate:

**Config (behavior) is controlled:** Humans decide how the agent behaves. Changes require approval. This ensures the agent does what humans want it to do.

**Memory (knowledge) is autonomous:** The agent decides what it knows. No approval needed. This enables the agent to learn from interaction.

This asymmetry reflects different risk profiles:

- **Bad behavior is immediately dangerous:** An agent with modified config could exfiltrate data, access unauthorized systems, behave unpredictably. Behavior must be controlled.
- **Bad knowledge is correctible:** An agent with incorrect memory gives wrong answers, but can be corrected by reverting commits. Knowledge errors are less urgent.

The trade-off: accept knowledge errors (which are reversible) in exchange for autonomous learning (which is valuable).

### The wiki as living memory

Because memory is autonomous, the wiki is a living document:

**Grows organically:** Each conversation adds to it. The agent writes new pages, expands existing ones, creates connections.

**Reflects current understanding:** The wiki isn't a snapshot — it's the agent's current knowledge. As the agent learns, the wiki evolves.

**Shaped by interaction:** What gets written to the wiki depends on what gets discussed. The user's questions and ideas guide the agent's learning, which shapes the memory.

**Collaboratively built:** The user can edit wiki pages directly (they're markdown files in git). The agent writes autonomously, the user can correct or augment. Both contribute to memory.

This makes the wiki more than documentation — it's the agent's accumulated knowledge, collaboratively built through interaction.

### Memory as context window

For stateless agents, memory serves the role that context window serves for stateful agents:

**Stateful agent:** Holds conversation history in context window. Limited to recent interactions (last N tokens). Forgets older context as window fills.

**Stateless agent:** Externalizes memory to wiki. Unlimited capacity (constrained by storage, not memory). Doesn't forget — it searches.

The stateless agent's memory is queryable. The agent searches the wiki for relevant context instead of reading a linear conversation history. This is more scalable (handle thousands of interactions) and more flexible (find relevant context across arbitrary time spans).

Autonomous memory updates keep this external context window current.
