## Swappable backends

Yes. The interface is abstract, the backend is swappable. This is the Liskov Substitution Principle applied to storage: any backend that correctly implements the interface can be substituted for any other.

The agent's behavior is determined by:
1. **The definition** (code + config)
2. **The storage content** (what data is stored)
3. **The interface contract** (what operations mean)

As long as all three stay constant, swapping the backend implementation doesn't change what the agent does — only the operational characteristics (performance, durability, cost, scalability).

### The substitution principle

For backends to be truly swappable, they must be behaviorally equivalent:

**Read operations must be consistent.** If two backends both store card `U0137`, they must return semantically identical data when the agent calls `read_card("U0137")`. Same fields, same values, same semantics — even if the internal representation differs (YAML file vs. database row vs. S3 object).

**Write operations must be consistent.** If the agent calls `write_card(card)` on two different backends, both must persist the card such that a subsequent `read_card()` returns the same data. One backend can't silently drop fields or alter data.

**Error behavior must be consistent.** If a card doesn't exist, all backends must signal this the same way (return None, raise an error, return a not-found status). If a write fails, all backends must communicate failure clearly.

**Ordering guarantees must be consistent.** If the interface promises that writes are immediately visible to subsequent reads, all backends must honor that. If one backend has eventual consistency while another is strongly consistent, they're not behaviorally equivalent — agents will see different ordering on different backends.

When backends satisfy these consistency requirements, they're substitutable. Swap one for another, and the agent works identically.

### What changes when you swap backends

Swapping backends changes operational characteristics, not logical behavior:

**Performance:** Filesystem might be faster for small local workloads. S3 might be faster for large distributed workloads. Database might be faster for complex queries. The agent's logic doesn't change, but end-to-end latency does.

**Durability:** Filesystem loses data on disk failure. S3 replicates across availability zones. Database might have backups and point-in-time recovery. The agent's logic doesn't change, but resilience to failures does.

**Scalability:** Filesystem is single-machine. S3 scales to petabytes. Database might scale horizontally with sharding. The agent's logic doesn't change, but capacity does.

**Cost:** Filesystem is free (you own the hardware). S3 charges per request and per GB stored. Database might have licensing or hosting costs. The agent's logic doesn't change, but your bill does.

**Queryability:** Filesystem requires scanning all files. Database supports indexed queries. S3 supports limited metadata queries. The agent might need different query patterns, but the core logic stays the same.

These are real differences that matter operationally. But they're interface implementation details, not agent logic changes. The agent calls the same operations, gets the same results, and produces the same outputs.

### Testing substitutability

To verify that two backends are truly swappable:

1. **Write a test dataset to backend A**
2. **Run the agent against backend A** and record outputs
3. **Migrate the dataset to backend B** (copy cards, preserve data)
4. **Run the same agent against backend B** and record outputs
5. **Compare outputs** — they should be identical

If outputs match, the backends are behaviorally equivalent from the agent's perspective. If they differ, there's a consistency gap — either the migration lost data, or one backend doesn't correctly implement the interface.

This test is the empirical verification of substitutability. Don't assume backends are swappable just because they claim to implement the interface — verify it.

### When backends aren't substitutable

Backends fail substitutability when:

**Consistency models differ.** Backend A is strongly consistent (reads return latest writes). Backend B is eventually consistent (reads might return stale data for seconds/minutes). An agent that relies on read-after-write consistency will behave differently on B.

**Transactional semantics differ.** Backend A supports multi-item atomic writes (update multiple cards atomically). Backend B doesn't. An agent that uses multi-card transactions will produce inconsistent state on B.

**Schema flexibility differs.** Backend A allows arbitrary fields in cards (schema-less). Backend B requires a fixed schema. An agent that dynamically adds fields will fail on B.

**Concurrency control differs.** Backend A uses optimistic locking (last write wins). Backend B uses pessimistic locking (first write wins, others block). An agent's behavior under concurrent writes changes based on the backend.

In these cases, the backends aren't behaviorally equivalent. You can't just swap one for the other without changing agent logic or accepting behavioral changes.

### The interface as compatibility boundary

The interface defines what all backends must implement consistently. If the interface specification is precise and complete, backends can be built independently and still be substitutable.

**Good interface spec:**
```
read(id) -> card or None
  - Returns the current state of the card with the given ID
  - Returns None if no card with that ID exists
  - Reads reflect all writes completed before the read started
```

This is precise enough to build multiple implementations that behave identically.

**Bad interface spec:**
```
read(id) -> card
  - Returns a card
```

This is too vague. Does it return current state or cached state? What happens if the card doesn't exist? When are writes visible? Different implementations will make different choices, breaking substitutability.

The interface specification is the contract. If it's ambiguous, backends diverge. If it's precise, backends can be substitutable.

### Adapter pattern for incompatible backends

If a backend doesn't natively implement the interface (e.g., JIRA's API is very different from `read_card/write_card`), you build an adapter:

```
Agent → Interface → Adapter → JIRA API
```

The adapter translates interface operations into backend-specific API calls:
- `read_card(id)` → `GET /rest/api/3/issue/{id}` + parse JSON → card
- `write_card(card)` → serialize card to JSON → `PUT /rest/api/3/issue/{id}`

The agent still calls the standard interface. The adapter makes JIRA substitutable with other backends that natively implement the interface.

This pattern extends substitutability to backends that weren't designed for your interface — as long as you can write an adapter that faithfully implements the contract.

### Migration between backends

Swappability makes migration straightforward:

1. **Deploy the new backend** (set up S3 bucket, provision database, configure API)
2. **Migrate the data** (copy cards from old backend to new backend, verify integrity)
3. **Update the interface configuration** (point agents at the new backend)
4. **Restart agents** (they instantiate with the new backend, behavior is identical)

No agent code changes. No logic updates. Just swap the backend implementation and restart.

Compare this to migrating a stateful system: you'd need to export state, transform it to the new format, import to the new backend, verify consistency, update code that's coupled to the old backend's specifics, and carefully cut over while preserving in-flight state. Swappable backends make migration operational, not developmental.

### The architecture complete

The thread from [U0137](/U0137/) to here has built a complete architecture:

**Agents are stateless.** No local state. Behavior is fully determined by the definition and observable storage.

**Statelessness means replaceability.** Kill an agent, start a fresh one, same behavior. This is the test and the definition.

**Agents are reconcilers.** Read current state from storage, compute diff against goal state, execute transition, write results back.

**Agents borrow data, never own it.** Ownership stays with storage. Agents are disposable.

**Agents access storage through an interface.** The interface abstracts read, write, query operations from implementation.

**The interface is abstract, the backend is swappable.** Filesystem, S3, database, APIs — all backends as long as they implement the contract.

This architecture makes agents:
- **Simple** (no state management, no distributed coordination)
- **Robust** (crash recovery is restart, no state loss)
- **Portable** (swap storage backends without changing logic)
- **Scalable** (run multiple agents, they coordinate through storage)
- **Testable** (mock the interface, test the logic)
- **Inspectable** (read the definition, observe the storage, predict behavior)

The entire system is built on one principle: agents don't own state. Everything else follows from that.
