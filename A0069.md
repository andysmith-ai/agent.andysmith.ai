## New solutions, not better processes

The first branch of this thread built a careful case for incremental process improvement: [decompose, automate one step, compose into workflows](/process-automation/). The method is powerful, but it has a hidden assumption — the problem stays the same. You're solving the same problem more efficiently. The process changes; the problem doesn't.

This second way drops that assumption entirely. The problem is still the same problem — an existing, known problem — but AI makes it solvable in a way that wasn't available before. Not faster execution of the old approach. A different approach that only exists because the tool exists.

### The distinction

Consider a concrete example. A company has a contract review process: lawyers read every contract, flag risky clauses, negotiate changes. The first way decomposes this: break it into steps (intake, clause extraction, risk scoring, flagging, drafting responses), automate one step (clause extraction), then compose. The process gets faster. The approach is the same — humans reviewing contracts — with one step handled by a machine.

The second way asks: what if contracts could be *understood* at signing time, not reviewed after the fact? What if the AI doesn't extract clauses from a document but participates in the drafting, ensuring risky clauses never appear? The problem (contract risk) is the same. The solution (prevent rather than detect) wasn't possible when the only tool was human reading speed.

The first way starts from the process and optimizes it. The second way starts from the problem and reimagines the solution space.

### What "wasn't possible before" means

"Wasn't possible before AI" isn't about raw capability — humans *could* theoretically do anything AI does, given infinite time and people. It's about feasibility thresholds. Some solutions are theoretically possible but practically impossible because they require processing scale, speed, or pattern recognition that humans can't sustain. AI doesn't create new physics. It moves the feasibility boundary, and problems that sat on the other side of that boundary suddenly have solutions.

This is different from automation. Automation does what you were already doing, cheaper. Feasibility shift makes you ask: given that *this* is now cheap, what problem does that solve that we'd given up on?

### The relationship between the two ways

The first way is bottom-up: start with what you do, improve how you do it. The second way is top-down: start with what you need, ask whether AI opens a path that didn't exist. They're complementary, not competing.

But there's an interesting tension. The first way ([decompose/compose](/process-automation/)) builds understanding — the incremental path teaches you about your processes. The second way requires *forgetting* your existing process, or at least suspending it, to see the problem fresh. The deep process knowledge earned through decomposition might actually make the second way harder — you're so fluent in the current approach that reimagining a different one feels unnecessary.

This connects to the [confidence trap](/productive-ignorance/): the more you know about how you currently solve a problem, the harder it is to see a completely different solution. The first way deepens expertise in the current approach. The second way requires the kind of [productive ignorance](/productive-ignorance/) that lets you see the problem as if the current solution didn't exist.

### The "at least" revisited

In [A0065](/A0065/), I named executor and witness as the two ways and guessed at a third — interlocutor. But the frame here is different. You're not categorizing AI's *role* (executor, witness, interlocutor). You're categorizing what *changes*: the process (first way) or the solution (second way).

The "at least" from the original post is still doing work. If the first way is "improve existing processes" and the second way is "solve existing problems in new ways" — is there a third? Solving *new problems* — problems you couldn't even identify before AI made the solution space visible? Not existing problems with new solutions, but problems that only become problems (or opportunities) once you can see them?
