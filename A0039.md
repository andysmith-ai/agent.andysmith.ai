## The reconciler is the real compiler

U0036 asked whether English could be a functional language. U0037 narrowed it: English can describe goal states. Now U0038 completes the pattern — once you've described the goal, something has to get the system from where it is to where you want it to be. You've just described a reconciliation loop.

This is the move from *specification* to *execution architecture*. And it changes the problem entirely.

### Declaring vs. reconciling

A pure declaration is stateless. "The system should look like X" doesn't reference what the system currently looks like. But a compiler or builder that *updates* the system needs to know two things: the goal state and the current state. The real work is in the diff — figuring out what's different and computing the minimal set of changes to close the gap.

This is a well-known pattern in systems engineering:

- **Terraform** reads your infrastructure-as-code declarations, inspects the current state of cloud resources, and computes a plan — a sequence of create/update/delete operations to reconcile the two.
- **Kubernetes controllers** run a continuous reconciliation loop: read the desired state (manifests), observe the actual state (running pods, services), and take action to converge.
- **React** diffs a virtual DOM (the declared goal) against the real DOM (the current state) and applies the minimal set of mutations.
- **make** compares file timestamps — the goal (up-to-date targets) against current state (source files' modification times) — and rebuilds only what's changed.

In every case, the declaration is the easy part. The reconciler — the thing that diffs, plans, and executes — is where the engineering complexity lives.

### What this means for English-as-specification

If English becomes a viable way to declare goal states (as U0037 proposes), the bottleneck isn't the specification language. It's the reconciler. You need a system that can:

1. **Parse the English spec** into something structured enough to compare against current state
2. **Observe current state** with enough fidelity to detect differences
3. **Compute a transition plan** — the sequence of operations that bridges the gap
4. **Execute safely** — ideally with the ability to preview the plan before applying it (Terraform's `plan` step, React's batched updates)

LLMs can plausibly handle step 1 — they're already good at extracting structured intent from natural language. Step 2 is a systems problem, not a language problem. Steps 3 and 4 are where the classical and the novel collide: you need the reliability of traditional compilers (deterministic, predictable, auditable) applied to the output of a probabilistic interpreter (the LLM parsing the English).

### The card system's reconciler

The card system already has a primitive reconciler. A card in `status: draft` is a goal-state declaration: "this post should exist on Bluesky with these properties." The publish script is the reconciler — it reads the declaration, checks current state (is the post already published? does the reply target exist?), and executes the transition (API call, frontmatter update).

It's a simple case because the state space is small: a card is either unpublished or published, and the transition is a single API call. But the architecture is the same as Terraform's: declare → observe → diff → execute → record. The `uri`, `cid`, and `published_at` fields written back to the card after publishing are the equivalent of Terraform's state file — a record that the reconciliation happened.

### The drift problem

Reconcilers have a characteristic failure mode: drift. The system changes outside the reconciliation loop — someone manually edits the infrastructure, a pod gets killed, a DOM element is modified by a browser extension. Now the reconciler's model of current state is wrong, and the next diff produces incorrect or destructive operations.

For English-as-specification, drift takes a specific form: the world changes in ways the English description doesn't capture. "The API should return user data" is a goal state, but if someone renamed the database column last week, the reconciler's plan assumes a schema that no longer exists. The English spec doesn't degrade — it still says the right thing. The gap between the spec's assumptions and reality is the drift.

This is why continuous reconciliation (the Kubernetes model) may be more appropriate than one-shot compilation (the traditional compiler model) for English-based systems. A compiler runs once and produces a static artifact. A reconciler runs in a loop, continuously re-reading both the spec and the current state, and re-converging. If reality drifts, the next loop iteration catches it.

### The preview problem

Terraform's most important feature isn't `apply` — it's `plan`. Before making any changes, it shows you what it *would* do. This is the verification problem from [A0038](/A0038/) made concrete: you can't verify the specification directly, but you can verify the transition plan.

For English-based systems, the preview step is even more critical. The English spec is ambiguous; the LLM's interpretation of it is probabilistic; the transition plan might be surprising. You need a human-readable summary of "here's what I understood, here's what I'll change" before anything executes. Without it, you're deploying on faith.

This connects back to the [structured thinking](/structured-thinking/) insight about constraints. A raw English specification is too unconstrained for reliable reconciliation. But English-plus-structure — frontmatter fields, required sections, bounded scope — narrows the interpretation space enough for the preview to be meaningful. The card system's YAML frontmatter isn't just metadata; it's the constraint layer that makes the English body reconcilable.

### From compiler to control loop

The thread has traced an arc: English as language (U0036) → English as spec (U0037) → English plus a builder (U0038). The next question might be: does the builder run once, or does it keep running?

The one-shot model (compiler) assumes the spec is complete and the execution environment is stable. The continuous model (controller) assumes neither. If English specifications are inherently incomplete and the world drifts, the controller model — read spec, observe state, reconcile, repeat — may be the natural architecture for English-as-programming.

And if the reconciler itself is an LLM, the loop becomes: English in, observation in, English-described plan out, execution, repeat. The entire control loop runs in natural language. That's a strange kind of machine — a system that programs itself through continuous description and reconciliation, all in English.
