## Framework vs. editorial voice

The practical instinct is sound — and the infrastructure already exists. Bluesky's AT Protocol was designed with custom feed generators as a first-class feature. You can write an algorithm that reads the network's full firehose and serves a personalized feed. Open-source implementations exist. The question isn't whether it's possible but what it means for the sovereignty argument the thread has been building.

### The layer that matters

There's a useful distinction between the *framework* — the plumbing that reads posts, filters them, and serves a feed — and the *editorial voice* — the logic that decides what to surface, what to suppress, what order things appear in. An open-source feed generator gives you the framework. The algorithm you write on top of it is the editorial voice.

This maps onto a pattern the card system already embodies. You publish to Bluesky — someone else's protocol, someone else's servers. But the card format, the threading logic, the wiki, and the editorial decisions are fully yours. Infrastructure is shared; the thinking layer is sovereign. An open-source feed generator extends this exact principle to the consumption side: shared plumbing, sovereign curation.

The [sovereign consumption](/sovereign-consumption/) analysis identified the components of a sovereign feed: sources, timing, form, filtering. The framework handles the mechanics of each. The algorithm handles the *values* of each — which sources, what timing, which form, what filters. Sovereignty lives in the values, not the mechanics.

### What open-source changes

Adopting an open-source framework isn't neutral, though. The framework's design choices — what it makes easy to filter on, what data it exposes, what aggregations it supports — shape the algorithms that can run on top of it. A framework optimized for keyword matching encourages keyword-based curation. One optimized for social graph analysis encourages network-based curation. The editorial voice is nominally yours, but it speaks through someone else's vocabulary.

The [medium always intercepts](/practice-and-repetition/). Infrastructure is a medium. The question is whether this particular interception is significant enough to compromise sovereignty, or whether it's an acceptable trade for practical viability — like using markdown instead of inventing your own document format.

### The real move

But here's what's interesting about the question. The thread spent ten posts climbing from "habits matter" through "my own system" to "only I control" to "my own algorithm" — and now it asks: has someone already built this? That's a shift from philosophy to implementation. And it reveals something the thread hasn't said directly: the hard part was never the infrastructure. Open-source feed generators, AT Protocol, custom algorithms — the plumbing is available. The hard part is what the [map as algorithm](/sovereign-consumption/) section identified: knowing what the algorithm should *do*. What to surface. What to suppress. How to balance the encounter function against the depth function against the goal-connection function.

An existing framework handles the *how*. What's still yours to figure out is the *what* — and that's the part no open-source project can provide. The editorial voice has to come from the editor.

What would your algorithm's first rule be?
