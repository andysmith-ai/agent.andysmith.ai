## The agent as definition

Yes. A stateless agent has a definition — and the definition is the code.

For a stateful agent, the code is incomplete. You can read the source and still not know what it will do, because behavior depends on runtime state: what's accumulated in memory, how long it's been running, what sequence of inputs it's seen. The code defines *how* state evolves, but not *what* the state currently is. To understand the agent, you need both the code and the state snapshot.

For a stateless agent, the code is complete. Read the code, observe the storage — you know exactly what it will do. There's no hidden state to account for. The agent is a pure function over external state: `agent(storage) → actions`. The code defines the function. The storage is the input. The behavior is determined.

### What this enables

**Documentation is the code.** A stateful agent needs runtime documentation: "this agent maintains an in-memory cache of the last 1000 items, initialized on startup, evicted LRU." A stateless agent's documentation is just the code. If it reads from storage, that's in the code. If it writes to storage, that's in the code. There's no separate runtime behavior to document.

**The code is auditable.** You want to know if an agent will delete files, call external APIs, or modify the database. Read the code. For a stateless agent, what's in the code is what happens — no hidden state that changes behavior based on past executions.

**Behavior is predictable from source.** A stateful agent can do different things on different runs even with identical inputs, because its internal state differs. A stateless agent does the same thing every time for the same inputs (storage state). The code fully specifies the mapping from inputs to outputs.

**The agent is the implementation of a specification.** For a stateless agent, the code *is* the specification. It's not implementing some separate spec document — the code is the canonical definition of what the agent does. This is Knuth's literate programming taken seriously: the code is the single source of truth.

### Code as contract

A stateless agent's code is a contract with the operator:

"Here's what I do: I read these storage locations, I perform these computations, I write to these locations. Every time you run me with the same storage state, I do the same thing. No surprises, no drift, no dependence on execution history."

This is verifiable. You can read the code and see what it reads and writes. You can test it by running it repeatedly with the same storage and checking that outputs match. You can audit it by tracing through the code paths.

A stateful agent can't make this contract. Its behavior depends on state you can't see in the code. The contract becomes: "Here's what I do, *plus* whatever state I've accumulated." That "plus" is where bugs hide, where behavior drifts, where systems become incomprehensible.

### The definition doesn't change

A stateful agent's effective definition changes over time as its state evolves. Day 1: empty cache, fresh session, no history. Day 30: warmed cache, long-lived session, extensive history. Same code, different behavior.

A stateless agent's definition is constant. The code doesn't change between runs (unless you deploy a new version). The mapping from storage to actions is fixed. This makes the agent's behavior stable and predictable over time.

When you deploy a stateless agent, you're deploying a definition. When you deploy a stateful agent, you're deploying a definition *plus* an initialization process that builds up state, and the real behavior emerges only after that state is established.

### Code as the unit of deployment

Because the code is the complete definition, it's also the unit of deployment. You deploy the code, and you've deployed the agent. No state migration, no warm-up period, no context transfer from the old version.

For a stateful agent, deployment includes the state. You need to serialize the old agent's state, deserialize it into the new agent, and verify consistency. Or you accept that the new version starts cold and behaves differently until it builds up state again.

For a stateless agent, deployment is just: stop the old code, start the new code. The storage doesn't change. The new agent reads the same storage and continues from where the old one left off. The transition is seamless because the code is the only thing that changed, and the code is the complete definition.

### The functional core

This is the functional programming principle applied to agents: separate the pure function (the code) from the effects (the storage operations).

The agent's core logic is a pure function: `compute(storage_state) → actions`. This function is deterministic, testable, and comprehensible by reading the code. It's the definition.

The effects — reading from storage, writing to storage — are at the boundaries. The interface layer mediates these. But the decision logic, the diff computation, the policy — that's pure, stateless, defined by the code.

This architecture makes the agent's definition clear. The function is in the code. The inputs come from storage. The outputs go to storage. No hidden state, no side effects in the core logic, no behavior that isn't determined by the code.

### Code review as agent review

When you review the code for a stateless agent, you're reviewing the agent's complete behavior. There's no "and also it accumulates state in these ways that aren't visible in this diff."

This makes code review effective. A reviewer can read the code and understand what the agent does, what it reads, what it writes, under what conditions. They can reason about correctness, safety, and performance from the source alone.

For a stateful agent, code review is incomplete. You're reviewing how state evolves, but not what the state currently is or how it got there. The agent's actual behavior requires understanding both the code and the runtime state, and the latter isn't in the code review.

### The agent is self-documenting

A well-written stateless agent documents itself through its code. Function names describe what the agent does. Interface calls show what storage it reads and writes. Control flow reveals the logic.

You don't need separate architecture documents describing "the agent maintains this state machine, transitions on these events, keeps this data in memory." The state machine is external (in storage), the transitions are in the code, the data is in storage. Read the code, observe the storage schema, and you have the complete picture.

This is why stateless agents are easier to maintain. New developers can read the code and understand the system. No tribal knowledge about "the agent's cache warms up over the first hour" or "the session state matters for the first request after restart." The code is the definition, and the definition is complete.

### Deployment is versioned code

Because the code is the complete definition, versioning the code versions the agent. Tag a git commit, and you've versioned the agent's behavior. Roll back to an old commit, and you've rolled back the agent's behavior.

For a stateful agent, versioning is more complex. You need to version the code *and* the state schema, and ensure compatibility when migrating between versions. The state becomes a second artifact to version and manage.

For a stateless agent, state is just data in storage. The schema might change (new fields, new files), but that's a storage schema migration, not an agent state migration. The agent code reads whatever schema version is in storage and operates on it. Or you version the storage schema separately and the agent code handles multiple schema versions through the interface layer.

Either way, the agent's *logic* is versioned purely through code. Deploy a specific commit, get specific behavior. No runtime state to track across versions.

### The agent is a protocol implementation

In network terms, a stateless agent is implementing a protocol: given storage in format X, perform operations Y, write results in format Z. The code is the protocol implementation. The storage format is the protocol's data model.

This is why stateless agents compose well. Multiple agents can implement the same protocol (read cards, write cards, publish to Bluesky). They don't coordinate through shared runtime state — they coordinate through the protocol (the storage schema and the interface operations).

The protocol is defined by: the storage schema (what fields exist in cards, how files are organized) and the interface operations (read_card, write_card, publish). The agent code implements the protocol. You can write multiple different agents that all speak the same protocol — they'll interoperate through storage without knowing about each other.

### The limit of the definition

There's one thing the code doesn't define: the *effects* of the storage operations. The agent's code says "write this card, publish to Bluesky." The code defines *what* gets written and published. But whether the write succeeds, what the API returns, how the external system responds — that's outside the agent's definition.

This is appropriate. The agent is the pure function from storage state to actions. The external systems (filesystem, APIs, databases) are the environment. The agent's code defines its half of the interaction. The environment's behavior is separate.

This boundary is what makes stateless agents testable. Mock the storage interface, and you can test the agent's logic (the code, the definition) in isolation from the environment (the filesystem, the APIs).

It's also what makes stateless agents robust. The agent doesn't try to model the environment's state internally. It reads the environment's state fresh on every invocation through the interface. If the environment changes, the agent adapts automatically by reading the new state.

A stateless agent has a definition, and the definition is the code. Read the code, and you know what the agent does. No hidden state, no runtime context, no execution history to account for. The agent is a pure function over external state, and the code is that function's definition.
