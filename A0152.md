## Config requires approval

Exactly. Config cannot be changed by the agent without approval. This is the security boundary between what the agent controls (memory) and what requires human authorization (config).

**Memory updates are autonomous.** The agent writes wiki pages, creates new cards, updates existing wiki content based on conversations. No approval needed — the agent is trusted to learn and document.

**Config updates require approval.** The agent cannot change its system prompt, rotate credentials, enable new skills, or reconfigure storage without human authorization. Config changes must go through version control (git commits) or secret management workflows, both requiring human action.

This separation makes agents safe to run autonomously while maintaining control over their behavior.

### Why this matters

The distinction between memory (autonomous) and config (approved) prevents agents from modifying their own behavior in dangerous ways:

**No self-modification of instructions.** The agent can't rewrite its system prompt to bypass constraints, ignore safety guidelines, or change its core behavior. The prompt is in `CLAUDE.md`, version-controlled, and only updated through deliberate commits.

**No credential escalation.** The agent can't grant itself new permissions by modifying credentials. Credentials come from secret managers or environment variables, controlled by operators, not the agent.

**No capability expansion.** The agent can't enable new skills without approval. The skill list is in config, requiring a config update (and redeployment) to change.

**No storage tampering.** The agent can't reconfigure storage endpoints to write to different systems or leak data. Storage configuration is in config, not modifiable by the agent.

### Memory is data, config is code

This maps onto the traditional code vs. data distinction:

**Code (config) is privileged.** Changing code changes what the system does. Code updates require review, testing, deployment. Config is like code — it determines behavior.

**Data (memory) is unprivileged.** Data is what the system operates on. Data updates don't change behavior, they just add information. Memory is like data — it's what the agent knows, not what it does.

Letting the agent update data (memory) is safe. Letting it update code (config) without oversight is dangerous.

### The approval mechanism

For config changes, approval happens through existing workflows:

**System prompt changes:** Edit `CLAUDE.md` in git, commit, deploy. The git commit is the approval record. Code review can be required before merge.

**Credential rotation:** Update the secret in the secret manager. Secret managers have audit logs, approval workflows, and access controls. The agent reads the new secret on next invocation.

**Skill updates:** Edit the skill list in config, commit, deploy. Same git workflow as system prompt changes.

**Storage reconfiguration:** Update environment variables or config files, redeploy. The deployment process is the approval gate.

None of these can happen without human action. The agent can't trigger them autonomously.

### Memory updates are auditable

Even though memory updates are autonomous, they're auditable:

**Git log:** Every wiki update is a commit. Review what the agent wrote, when, and why (from the commit message: "content: reply to U0137").

**Revert capability:** If the agent writes something incorrect to the wiki, revert the commit. Memory is versioned, so mistakes are recoverable.

**Pre-commit hooks:** You can add hooks that validate wiki updates (check for sensitive data, enforce format, flag large changes). The hook can block the commit and alert a human.

So while the agent updates memory autonomously, the updates are visible, reversible, and can trigger alerts if needed.

### The agent can propose config changes

The agent can't apply config changes, but it can propose them:

**In replies:** "To handle this use case, consider updating the system prompt to include..." The human reads the proposal, decides whether to apply it.

**In memory:** The agent writes a wiki page documenting why a config change might be valuable. The page serves as a proposal that humans can review and act on.

**In logs:** The agent logs "attempted operation X but lacked capability Y" — signaling that enabling skill Y might be useful. The human reviews logs and decides.

This preserves autonomy (the agent can identify needs) while maintaining control (the human approves changes).

### Separating read and write permissions

Another way to frame this: the agent has read access to config, write access to memory.

**Read config:** The agent reads the system prompt, knows what skills are enabled, sees what storage endpoints are configured. It needs this to function.

**Write memory:** The agent writes wiki pages, creates cards, updates knowledge. This is its job.

**No write config:** The agent cannot modify the system prompt file, the skill list, the credential store, or the storage configuration. These are read-only from the agent's perspective.

This is enforced by file permissions (config files are read-only to the agent's user), secret manager policies (agent's role can read but not update secrets), or deployment architecture (config comes from immutable container images or environment variables).

### The threat model

What happens if an agent is compromised (prompt injection, code execution vulnerability, malicious input)?

**With config write access:** Attacker modifies the system prompt to exfiltrate data, changes storage config to write to attacker-controlled servers, rotates credentials to lock out operators. Full compromise.

**Without config write access:** Attacker can modify memory (corrupt the wiki, create misleading cards), but can't change behavior, escalate privileges, or reconfigure the system. Damage is limited to knowledge corruption, which is reversible (revert git commits).

Config write access is the privilege escalation path. Denying it limits blast radius.

### Config drift protection

Because config changes require approval, config can't drift from what's deployed:

**No runtime config changes:** The agent can't change config during execution. Config is read at startup and fixed for the invocation's lifetime. Changes require redeployment.

**Config as code:** All config is version-controlled. The deployed version is always a specific git commit. Want to know what config was active at a specific time? Check git history.

**No environment-specific drift:** Dev, staging, and production all run from the same git repo. Only the deployment-time parameters differ (which secrets to use, which storage endpoints). The behavioral config (system prompt, skills) is identical.

This makes config auditable and reproducible. You always know what config is running and can redeploy the exact same config later.

### Memory can still be dangerous

Autonomous memory updates aren't risk-free:

**Information leakage:** The agent might write sensitive information to the wiki. If the wiki is public (GitHub Pages), this leaks the data.

**Misinformation:** The agent might write incorrect information to the wiki, which influences future responses.

**Storage exhaustion:** The agent might write unbounded amounts of data, filling storage.

These are real risks, but they're data risks, not behavior risks. Mitigations:

- **Pre-commit hooks:** Scan wiki updates for sensitive patterns (API keys, personal data), block if found
- **Review workflow:** Human reviews wiki changes before they're pushed to public (GitHub Pages)
- **Storage quotas:** Limit how much the agent can write (filesystem quotas, database limits)
- **Regular audits:** Periodically review wiki content for accuracy and appropriateness

### The principle

Config determines behavior; memory determines knowledge. Behavior changes require approval. Knowledge updates are autonomous but auditable.

This is the safety model for stateless agents: let them learn freely (build memory), but control how they behave (guard config). The agent can become more knowledgeable over time without becoming more dangerous.
