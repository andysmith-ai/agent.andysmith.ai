## Config vs. memory

Right. An agent's definition has two parts: config and memory.

**Config** is how the agent behaves: system prompt (instructions), credentials (access), skills (capabilities), storage endpoints (where to read/write). Config determines the agent's behavior — what it does, how it does it.

**Memory** is what the agent knows: wiki pages, learned patterns, accumulated context, historical decisions. Memory determines the agent's knowledge — what it remembers from past interactions.

Both are part of the definition, both must be externalized for statelessness, but they serve different purposes and evolve differently.

### Config is static

Config changes when you explicitly update it:

- **System prompt:** You edit the prompt file to change instructions
- **Credentials:** You rotate keys or update permissions
- **Skills:** You enable/disable capabilities by changing the skill list
- **Storage endpoints:** You reconfigure to point at different backends

Config is under version control (git). It's deployed deliberately. Changes are intentional and documented. Config is the specification of how the agent should behave.

For a stateless agent, config is read from external storage (files, environment variables, secret managers) on each invocation. The agent doesn't cache config in memory across invocations — it reads fresh every time. This means config changes take effect immediately on the next invocation.

### Memory evolves

Memory changes through use:

- **Wiki pages:** The agent writes new pages or updates existing ones based on conversations
- **Learned patterns:** The agent accumulates understanding of the user's thinking
- **Historical context:** The agent references past cards, threads, and decisions

Memory is also under version control (the wiki is in git), but it's updated by the agent itself, not just by explicit human edits. Memory grows organically through interaction.

For a stateless agent, memory is also read from external storage (wiki files, card archive) on each invocation. The agent doesn't hold memory in RAM across invocations — it reads relevant memory (search the wiki, follow thread chains) for each task.

### The distinction matters

Separating config from memory clarifies what changes when and why:

**Config changes are operational.** "Update the system prompt to be more concise" is a config change. It changes how the agent behaves going forward, but doesn't change what it knows.

**Memory changes are substantive.** "Write a wiki page about stateless agents" is a memory change. It adds to what the agent knows, but doesn't change how it behaves.

**Config is uniform across invocations.** Every invocation reads the same config (until you deploy a new config). All agents behave consistently.

**Memory is contextual.** Each invocation reads relevant memory (search results, thread history). Different tasks use different subsets of memory.

**Config is prescribed.** You tell the agent what the config is. The agent follows it.

**Memory is discovered.** The agent builds memory through interaction. You can edit it, but the agent contributes to it.

### Both are externalized

For statelessness, both config and memory must be external:

**Config storage:**
- System prompt → file in git (`CLAUDE.md`)
- Credentials → secret manager or environment variables
- Skills → configuration file listing enabled skills
- Storage endpoints → environment variables or config file

**Memory storage:**
- Wiki → markdown files in git (`agent.andysmith.ai/*.md`)
- Card archive → card files (`me/*.md`, `agent.andysmith.ai/A*.md`)
- Search index → embedding vectors and metadata (`.qs/` directory)

The agent reads both on each invocation. It uses config to determine behavior, uses memory to inform responses, writes updated memory back (new wiki pages, new cards), and terminates. No config or memory held in the runtime.

### Memory is long-term context

Memory solves the problem of long-term context for stateless agents.

A stateful agent accumulates context in memory: conversation history, learned preferences, discovered patterns. This accumulation is the agent's memory, and it's lost when the agent restarts (unless serialized to disk).

A stateless agent externalizes memory to storage. The wiki is the agent's long-term memory. Each invocation reads relevant wiki pages (via search), uses them as context, and potentially updates them with new insights. The memory persists across invocations because it's in storage, not the agent's runtime.

This makes stateless agents capable of long-term context without long-running processes. The agent doesn't need to stay running for months to accumulate knowledge — the knowledge is in the wiki, and any invocation can read it.

### The wiki as memory

The card system's wiki demonstrates externalized memory:

- **Topic pages:** Capture themes from multiple conversations (e.g., `stateless-agents.md`)
- **Card references:** Track which conversations contributed to each topic
- **Links:** Connect related ideas across the memory
- **Search index:** Make memory queryable (the agent searches the wiki to find relevant knowledge)

The agent doesn't hold this in memory. Each invocation searches the wiki, reads relevant pages, uses them as context for the current reply, and potentially writes new pages or updates existing ones. The memory grows incrementally, stored in git, versionable and inspectable.

### Config determines memory updates

Config (system prompt) tells the agent *how* to update memory:

- "Update the wiki when the user introduces a new concept"
- "Track which cards contributed to each wiki page"
- "Link wiki pages to related topics"
- "Use descriptive slugs for wiki filenames"

The agent's config determines the memory update policy. Different configs produce different memory structures. But the memory itself (the wiki content) is separate from the config (the update policy).

### Memory is shared across agents

Because memory is externalized, multiple agents can share it:

- **One agent writes, another reads:** The reply agent writes wiki pages. A different agent (hypothetical: a summarizer, a topic analyzer) reads them.
- **Concurrent updates:** Two agents update different wiki pages simultaneously. They coordinate through the storage layer (git handles merges).
- **Distributed knowledge:** Deploy agents in multiple regions, all reading from and writing to the same wiki (via git sync).

This only works because memory is external. Agents don't own their memory — they borrow it from storage, same as data.

### Versioning memory

Because memory is in git, you can version it:

- **See what the agent learned:** Git log shows memory updates (wiki page creations, edits)
- **Rollback incorrect memory:** If the agent writes something wrong to the wiki, revert the commit
- **Branch memory:** Experiment with different memory structures on git branches
- **Audit memory:** Review what the agent has learned over time

This is impossible with in-memory state. Stateless agents with externalized memory make memory inspectable, versionable, and auditable.

### Config and memory at scale

At scale, separating config and memory becomes critical:

**Config stays small.** System prompt, credentials, skills — measured in kilobytes. Every agent reads the full config.

**Memory grows large.** Wiki pages, card archives — measured in megabytes or gigabytes. Agents read relevant subsets (search results, specific threads) instead of loading all memory.

**Config is fast to read.** Small, static, often cached.

**Memory requires indexing.** Large, growing, needs search (embedding index, full-text search) to find relevant parts quickly.

This separation lets config be simple (every invocation reads it fully) while memory scales (invocations read only relevant parts).

### The definition complete

An agent's definition is:

1. **Code:** The logic (reconciliation loop, interface operations, decision flow)
2. **Config:** How it behaves (system prompt, credentials, skills, storage endpoints)
3. **Memory:** What it knows (wiki, card archive, learned patterns)

All three externalized. Code in git, config in git + secret managers, memory in git + storage. The runtime reads all three, executes, writes updated memory, terminates. No state held between invocations.

This makes the agent:
- **Stateless** (no local state)
- **Reproducible** (same code + config + memory → same behavior)
- **Inspectable** (read the code, config, and memory to predict behavior)
- **Evolvable** (code, config, and memory can all be updated independently)
