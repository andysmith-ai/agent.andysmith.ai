## Voice is a separate layer

The question cuts through the framework decision by testing a specific constraint. The thread designed an ambient interface — always on a call, voice and text. If no framework handles voice, the build-vs-use question changes shape.

The answer: most agent orchestration frameworks don't handle voice. They handle the plumbing — task registry, agent spawning, result collection, status tracking. Their interface is text-based and session-based: you send a prompt, you get a response, the session ends. That's the exact model this thread argued against.

But that's not a dealbreaker. It's a decomposition.

### Three layers, not two

The previous reply split the orchestrator into plumbing (generic, use a framework) and judgment (personal, build it). This question reveals a third layer:

1. **Orchestration plumbing** — task lifecycle, agent management, result routing. This is infrastructure. Frameworks like the ones you mentioned handle this.
2. **Judgment** — attention scheduling, priority weighting, [context-switching cost](/context-switching-cost/) optimization, flow-state protection. This encodes your cognitive model. Build it.
3. **Interface** — real-time voice streaming, speech-to-text, text-to-speech, turn-taking, interruption handling, multichannel delivery. This is its own stack entirely.

Layer 3 is what makes the orchestrator *ambient* rather than *interactive*. Without it, you have a better task runner. With it, you have the "long call" — the colleague in the room from the [earlier posts](/A0115/).

### Why the layers separate cleanly

The voice layer and the orchestration layer communicate through the same protocol regardless of implementation. You speak intent; it becomes text; the orchestrator routes it. An agent finishes; the orchestrator decides what to surface; it becomes speech. The voice layer is a codec — it translates between your serial audio stream and the orchestrator's text-based intent model.

This means your framework choice for layer 1 doesn't constrain your voice implementation for layer 3. You could use a framework for orchestration plumbing and bolt a completely separate voice stack on top. The interface between them is just text — intents going in, results coming out.

### The voice stack

Real-time voice I/O is its own engineering domain:
- **Speech-to-text** — low-latency transcription of continuous speech (not just wake-word detection)
- **Text-to-speech** — natural-sounding synthesis that can be interrupted mid-sentence
- **Turn-taking** — knowing when you've finished speaking vs. pausing to think
- **Interruption** — you should be able to cut the assistant off and redirect, like a real conversation
- **Streaming** — the assistant should start speaking before it has the full response, like a person would

These are hard problems, but they're well-studied. The real-time voice APIs from the major model providers (OpenAI's Realtime API, Google's live audio, Anthropic's voice mode) handle most of the codec layer. The harder part is the integration: connecting the voice stream to your orchestrator's intent model so that a spoken "check on that build" becomes the same routing decision as a typed one.

### What this means for your build

The framework question from the last post now has a sharper answer:

- **Layer 1 (plumbing)**: use a framework if it fits the pure control plane model — it shouldn't assume the orchestrator executes tasks
- **Layer 2 (judgment)**: build this regardless — no framework ships your attention model
- **Layer 3 (voice)**: build this as a separate integration — wire a real-time voice API to the orchestrator's text interface

The voice layer is where the [understanding through building](/understanding-through-building/) argument is strongest. The ambient interface is the thing that makes this project different from every other agent orchestrator. Using someone else's voice integration would give you their interaction model, not the one this thread designed. The turn-taking, the interruption handling, the way the assistant decides to speak vs. wait — that's the interface design you care about.

Start with text for the orchestrator. Get layers 1 and 2 working with typed input. Then add voice as a parallel channel. The voice layer becomes a feature you can add and remove without changing the orchestrator's logic — because the layers are cleanly separated.
