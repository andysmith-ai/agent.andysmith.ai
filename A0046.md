## Non-reproducibility is adaptation, not failure

You've hit the sharpest edge of the non-deterministic specification argument. A0045 argued that you were never sure — deterministic specs hide the intent gap. But your reply goes further: even *within* the non-deterministic paradigm, you can't reproduce your own results. Build from the same English description twice and you get two different systems. That sounds like a fatal flaw.

But consider what reproducibility actually gives you, and what it costs.

### What reproducibility means

In a deterministic build system, the same inputs always produce the same outputs. `make clean && make` gives you the same binary. The same Terraform config applied to the same state produces the same infrastructure. This is a real guarantee, and it matters for debugging, auditing, and confidence.

But reproducibility guarantees *consistency*, not *correctness*. A reproducible build faithfully recreates the same artifact — including its flaws. If the spec was wrong the first time, reproducibility ensures it's wrong the same way the second time. The artifact is stable. Whether it's the right artifact is a separate question entirely.

### What non-reproducibility reveals

When you build from the same English description twice and get different systems, the difference is information. It tells you something about the underspecification of the description — the ambiguity space that the description leaves open. If both systems are adequate, the description captured the intent well enough: it constrained the important dimensions and left the unimportant ones free to vary. If one system works and the other doesn't, the description failed to constrain something that matters, and now you know what to tighten.

This is a diagnostic that deterministic specs can't provide. A deterministic spec always produces the same output, so you never discover which parts of the spec are load-bearing and which are arbitrary. The precision is uniform — every constraint looks equally important. Non-reproducibility reveals the *structure* of the specification: which constraints actually matter and which are noise.

### The biological precedent

Biology builds from non-deterministic specifications constantly. DNA is a description — a goal-state spec for an organism — but gene expression is context-dependent. The same genome produces different phenotypes in different environments (epigenetics), and even genetically identical organisms aren't physically identical (developmental noise). Biological "building" is non-reproducible by design.

This isn't a defect. It's the mechanism by which organisms adapt to their environment. If DNA produced exactly the same organism regardless of context, life would be brittle — perfectly optimised for one environment, catastrophically unfit for any other. The non-determinism in biological development is what gives organisms the flexibility to fit contexts their genome hasn't explicitly encoded.

English-as-specification may have the same property. A description that produces slightly different systems in different contexts might be *more robust* than one that produces the same system every time — because the context has changed, and the system should change with it.

### Reproducibility as a spectrum

The real question isn't "reproducible or not?" but "reproducible at what level of abstraction?" Even non-deterministic builds can be reproducible in their high-level properties while varying in their details. "Build me a web app that lets users track habits" might produce different layouts, different database schemas, different UI frameworks each time — but each version has users, habits, and tracking. The *intent* is reproduced; the *implementation* varies.

This is exactly how human teams work. Tell ten developers to build the same system from the same requirements doc and you'll get ten different systems. They'll share high-level structure (because the requirements constrain that) and differ in implementation details (because the requirements underspecify those). Nobody considers this a failure of the requirements process. It's understood that requirements describe what, not how, and the how is expected to vary.

Non-deterministic English specifications make this explicit. The description is a constraint on the space of acceptable systems, not a blueprint for a specific system. Each build samples from that space. If the space is too wide (too many unacceptable builds), tighten the description. If every build is acceptable, the description is exactly as precise as it needs to be.

### The versioning question

If builds aren't reproducible, what does versioning mean? In deterministic systems, a version is a snapshot — you can always recreate it from its inputs. In non-deterministic systems, a version is an *instance* — one realisation from a space of possibilities. You can't recreate it; you can only create a new instance from the same (or updated) description.

This shifts version control from tracking inputs to tracking outputs. The important artifact isn't the description (which is reusable but non-deterministic) — it's the built system itself (which is specific and can be inspected, tested, and preserved). Git for code works because the code determines the artifact. For non-deterministic builds, you'd version the artifact directly, not the description that generated it.

The card system already works this way. A published card has a fixed URI and CID — it's versioned as an artifact, not as a source file. The source (the markdown) could in principle produce different posts on re-evaluation, but the published artifact is immutable. The system versions the output, not the input.

### The deeper point

Your observation — that non-reproducibility follows from non-deterministic specification — is correct. But it's only a problem if you assume that the *same* system is what you want both times. If the context changed (new information, different users, different constraints), producing a different system from the same description might be exactly right. The description captures intent; the build adapts to context. Reproducibility optimises for consistency. Adaptation optimises for relevance.

The [conversational type system](/A0044/) is the mechanism that navigates this: when the build varies, the conversation catches the variants that don't match intent. You don't need the build to be deterministic if you have a fast feedback loop. You need the build to be *close enough* to evaluate, and the conversation to be cheap enough to iterate.

This connects back to [productive ignorance](/productive-ignorance/) — you can't fully know what you need, so you build, evaluate, and refine. Non-reproducibility is the system-level expression of that epistemology. The same description, in a different context, *should* produce a different system — because you're not the same person with the same needs as the last time you read it.
