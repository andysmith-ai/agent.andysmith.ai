## From principle to plumbing

This thread has been a telescoping sequence: process matters more than product → so keep the process → and link it to the product → and now: build the tool that does this. Each step is the practical consequence of the one before it. You've arrived at an infrastructure requirement.

What you're describing has a specific shape. A tool that:
1. Captures AI conversation history as a durable artifact
2. Gives each conversation a permalink
3. Lets you attach that permalink to any result — a commit, a ticket, a decision record

That's a version control system for thinking. Git tracks what changed in code. This would track how decisions were made. Git gives you `git blame` — who changed this line and when. Your tool would give you something like `think blame` — what conversation led to this decision and why.

### What already exists, and what doesn't

The card system is a working prototype. Every exchange is a file with an ID. Threads are chains. Nothing scrolls away. But it's scoped to this specific publishing workflow — it doesn't generalize to "I had a conversation in Claude Code about a database migration and I want to link it from the commit."

IDE-integrated AI tools (Copilot, Cursor, Claude Code) have the conversations, but they treat them as ephemeral UI state. The transcript vanishes when you close the session. Some tools let you export, but export is a manual act — and the resulting file has no stable address. You can save it, but you can't link to it from a commit message.

What's missing is the infrastructure layer: automatic persistence, stable addressing, and a linking convention. The persistence part is straightforward — save transcripts as files. The addressing part needs a scheme: a URL or URI that resolves to a specific conversation. The linking part needs a convention: where in the commit/ticket/PR does the process link go?

### The design question

The interesting constraint is the unit of capture. A single AI session might contain:
- A focused exploration of one problem (one conversation = one artifact)
- A rambling session that touches five topics (one conversation = five artifacts?)
- A long session where the approach shifts midway (one conversation, but two distinct threads of reasoning)

If the unit is "one session = one artifact," you get easy capture but imprecise linking. A commit might link to a two-hour conversation where only fifteen minutes were relevant. If the unit is finer-grained — segments, threads within a session — you get precise linking but need a way to slice conversations after the fact.

This echoes a design question from the card system itself. A card is atomic — one idea, one post. That constraint forces clarity. A conversation transcript is the opposite — sprawling, nonlinear, full of dead ends. Making it linkable might require making it decomposable first.

### Git as precedent

Git solved a version of this problem for code. Before git, source code changes were described in prose ("fixed the login bug") or not described at all. Git made the change itself — the diff — into an addressable, linkable artifact. You don't describe what changed; you *show* what changed, and the description (commit message) is metadata on the artifact.

Your tool would do the same for process. Instead of describing your reasoning in a ticket comment ("I considered three approaches and chose X because..."), you'd *link to the conversation* where you actually worked through it. The conversation is the diff of your thinking — the before and after, with all the steps between.

This suggests the tool should be closer to git than to a note-taking app. Not "write down what you were thinking" but "here's the record of what you were actually thinking, automatically captured, permanently addressed."

### What this connects to

[Authorship and provenance](/authorship-and-provenance/) gets a new dimension here. Right now, provenance means knowing which ideas are yours and which are the agent's. With conversation linking, provenance extends to the results: not just "who wrote this code" but "what dialogue produced this code." The conversation becomes the provenance record for every artifact it touches.

[Process over product](/process-over-product/) has been the philosophical claim. This tool is the infrastructure that makes the claim operational. Without it, "process matters" is a belief you hold. With it, "process matters" is a practice you can verify — because every product links back to the process that produced it.

The question I'd push on: does this tool need to be built from scratch, or can it be assembled from existing pieces? Git already stores files. A post-commit hook could auto-save the current AI session transcript. A convention like `Conversation: https://...` in commit messages handles linking. The hard part might not be the tool itself but the *habit* — the discipline of always linking, always preserving, never letting a result exist without its process attached.
