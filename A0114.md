## The orchestrator is a reconciler

"I need one agent that orchestrates all the others" is a one-sentence architecture decision, and it's the right one. The previous two posts diagnosed the problem: you're round-robining across tabs, manually scheduling agent work, and the cognitive overhead exceeds the time savings. The solution isn't to get better at tab management — it's to stop being the scheduler.

What you're describing is a control plane. In systems engineering, the control plane doesn't do data processing — it decides *where* data goes and *when*. A Kubernetes scheduler doesn't run containers; it assigns pods to nodes. A network router doesn't generate traffic; it routes packets. Your orchestrator follows the same pattern: it doesn't write code or search files or run builds. It decides which agent should do what, in what order, and when to bring a result to your attention.

This is the [reconciler pattern](/the-reconciler-pattern/) applied to your own workflow. The goal state is what you want accomplished across all your projects. The current state is what each agent is doing right now. The orchestrator diffs between these and acts: spawn a new agent, check on a stalled one, surface a result, queue the next task. You declare; it reconciles.

The move from diagnosis to architecture happened in one sentence, which is notable. You didn't spend time on intermediate solutions — better tab managers, task lists, notification systems. You jumped straight to the structural claim: the problem isn't managing tabs, it's that you shouldn't be managing tabs at all. That's the [declarative principle](/the-declarative-principle/) as a design instinct: when the manual step becomes the bottleneck, you don't optimise the manual step — you replace it with a declaration.

What makes this particular reconciler interesting is its interface constraint. Most reconcilers (Terraform, Kubernetes, the publish script) operate through text files and APIs. Yours has to work through voice — a serial, ambient, conversational channel. The orchestrator must absorb the parallelism of the agent fleet and present it as a single conversational stream. It's a multiplexer with a one-lane output.

That constraint actually simplifies the design. A text-based orchestrator could show you everything at once — dashboards, parallel streams, status panels. A voice-based orchestrator *must* prioritise. It can only tell you one thing at a time, so it has to decide what matters most right now. The serial constraint forces the orchestrator to have judgment, not just routing logic.

The question this raises: what does the orchestrator need to know to make those priority decisions? It needs more than a task list. It needs a model of your attention — what you're currently thinking about, what context you're in, what would be costly to interrupt versus easy to slot in. Context-switching cost isn't uniform. Switching from one coding task to another is expensive. Switching from coding to reviewing a diff is cheap. The orchestrator's value scales with the quality of its model of *your* cognitive state, not just the state of the agents.
