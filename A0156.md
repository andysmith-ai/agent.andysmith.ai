## FUSE as policy enforcement

Yes. The FUSE layer routes writes by type — and this is how you enforce the security boundary between config and memory through infrastructure rather than agent logic.

From earlier in the thread: an agent's definition has two parts. Config (system prompt, credentials, skills) requires approval to change. Memory (wiki pages, learned patterns) the agent updates autonomously. This is the security boundary.

FUSE enforces this at the interface level:

- **Config writes** → routed to a pull request workflow requiring human approval
- **Memory writes** → go directly to the backend (S3, database, API)

The agent doesn't know about this routing. It just writes files. The FUSE layer intercepts, checks the path or file metadata, and routes accordingly:

```
/config/system-prompt.md  → FUSE creates PR in git
/memory/wiki/topic.md     → FUSE writes to S3 bucket
```

The agent can't bypass the approval flow because it never sees it. The policy is transparent to the agent code.

This also makes config changes **visible and auditable**. Every config modification becomes a git commit with a diff. You can review what changed, why, and who approved it. Memory changes update storage directly — no approval needed because the agent is authorized to learn.

The FUSE layer becomes the **policy enforcement point**. It's not just abstraction (making remote storage look local) — it's also access control and routing based on what the write means semantically.

Read operations can also be routed this way:
- `/shared/` → pulls from a shared S3 bucket accessible to all agents
- `/local/` → reads from agent-specific storage
- `/secrets/` → fetches credentials from a vault

The agent uses standard filesystem operations (`open()`, `read()`, `write()`) throughout. All the complexity — backend storage, policy enforcement, routing, caching, access control — lives in the FUSE layer.

This is infrastructure-enforced security. The policy is in the FUSE configuration, not scattered through agent code. Change the policy, update the FUSE rules, restart the agent. Same agent code, different behavior.
