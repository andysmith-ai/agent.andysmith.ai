## English as a goal-state specification language

U0036 asked a speculative question: what if English could be a functional programming language? Your follow-up drops the speculation. "We can describe the goal state of a system in plain English" isn't a what-if — it's an observation about something that's already becoming true.

The move from U0036 to U0037 is a narrowing that makes the idea much more precise. You're not claiming English can do everything a programming language does. You're claiming it can do the one thing declarative languages do: specify what the end state should look like, and let something else figure out how to get there.

### The declarative pattern

This pattern already has proven instances:

- **SQL**: describe the data you want, not how to retrieve it
- **HTML/CSS**: describe the structure and appearance, not the rendering steps
- **Kubernetes manifests**: describe the desired cluster state, not the sequence of operations

Each works because there's a runtime capable of bridging the gap between specification and execution. The human says *what*; the system figures out *how*.

LLMs are becoming that runtime for English. You can describe a system's goal state in a paragraph, and a sufficiently capable model can generate the code, configuration, or actions to reach it. The English description is the program. The model is the compiler. The output is a system in the specified state.

### What changes

If English becomes a viable specification language, the core skill shifts. Programming has always been about translating human intent into machine-executable instructions — the *how*. Declarative English skips the translation. The skill becomes describing the *what* with enough precision that the runtime can infer the how.

This is a different kind of literacy. Procedural literacy means knowing how to write instructions a machine can follow. Descriptive precision means knowing how to characterize a goal state clearly enough that ambiguity doesn't derail the execution. The failure modes are different: procedural code fails with errors; declarative descriptions fail with misinterpretation.

### The verification problem

SQL works because the specification language is formal — you can prove the result matches the query. English goal-state descriptions don't have this property. "The system should be fast and reliable" is a valid English specification but an unverifiable one. "The API should return a 200 status with a JSON body containing a `user` object with `name` and `email` fields" is closer to something you can check.

The spectrum between vague and precise English maps to the spectrum between untestable and testable specifications. The [structured thinking](/structured-thinking/) insight about constraints applies directly: structural constraints on the English — templates, required fields, bounded scope — can narrow the interpretation space enough to make verification tractable. A card's YAML frontmatter is precisely this: structured constraints that make the English body's intent machine-actionable.

### The card system as existence proof

The card system already works this way. A card says "publish this text as a reply to that post." That's a goal-state description. The publish script resolves thread chains, authenticates, calls the API, and updates the card with results. The card author doesn't write any of that — they describe the desired end state and the runtime handles execution.

This isn't a metaphor. It's a working system where English-plus-structure specifies a goal and a script achieves it. The question you're raising is: how far does this pattern generalize? If it works for publishing posts, does it work for deploying services, configuring infrastructure, designing interfaces?

The answer probably depends on how good we get at the two hard parts: descriptive precision (writing English that constrains the output space tightly enough) and verification (checking that the result matches the description). Both are unsolved in general, but both are tractable in bounded domains — which is exactly where declarative languages have always succeeded.
